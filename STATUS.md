# Asteroid Field Deluxe — STATUS

Last updated: 2026-07-06 · Build version: **1.8** (v2.0 Phase 7 of 9 shipped) · Last session: implemented **F7 — Basic Gamepad Input**. Refactored the input layer so each of `input.left`/`right`/`thrust`/`fire`/`shield` resolves *either* keyboard *or* the first connected gamepad, with **no call-site changes** anywhere — everything downstream still calls `input.left()` etc. exactly as before. The routing is a data-driven **`bindings`** table (one entry per action: keyboard `keys`, gamepad `buttons`, an optional analog `axis` `{index,dir}`, and a `fixed` rebindable flag), so Phase 8's rebinding UI has a single structure to build against. `pollGamepad()` runs once per frame in the main loop before `update()`: it reads `navigator.getGamepads()`, takes the **first connected** pad (skips `null` slots, ignores any others), and snapshots buttons/axes + the prior frame's buttons for edge detection. Held-state gameplay actions resolve lazily inside `input.*()` via `actionHeld`/`kbActive`/`gpActive`; menu/system actions (Start/A/B) are **edge-triggered** via `gpPressed` in a `handleGamepadMenu()` that mirrors the keydown handler. Defaults per F7 spec: D-Pad / Left Stick past a 0.25 deadzone for rotate + thrust, A to fire; **shield → right trigger *or* right bumper (a gap-fill — F7 didn't specify a controller shield button)**; Start = start/pause/unpause (toggles pause exactly like `P`), A = confirm, B = back (reserved — no menu to back out of until F8). **Hardcoded defaults only; the rebinding UI is Phase 8 (F8).**

## Working / verified

**Headless-verified this session (v1.8, `scratchpad/test-f7.js`, 52 assertions):**
- **Keyboard path unchanged** — with no gamepad connected, every predicate mirrors the keyboard exactly (ArrowLeft/A→left, D→right, ArrowUp/W→thrust, Space→fire, Shift→shield), and predicates still work when `pollGamepad()` was **never called** (the initial gamepad state) — this is the state the f2–f6 tests run in, so they stay green.
- **Default gamepad buttons resolve** — D-Pad Left/Right/Up → left/right/thrust; A → fire; Right Trigger and Right Bumper each → shield; the B button maps to *no* gameplay action (it's menu-only `back`).
- **Left-stick axis + 0.25 deadzone** — LX −0.5→left, +0.5→right, LY −0.5→thrust, LY +0.5 (stick down)→*not* thrust; the deadzone is a strict `>`, so −0.20 and exactly −0.25 don't register but −0.26 does.
- **Keyboard OR gamepad** — either source alone triggers an action; a connected-but-idle pad plus no keys leaves everything false; keyboard still works while a pad is connected.
- **First connected pad only** — with two pads, `pad[0]` wins and `pad[1]` is ignored; a `null` slot 0 is skipped so the first *non-null* pad is used.
- **Edge-triggered menu reads** — `gpPressed(pause)` is true only on the frame Start is first pressed (false while held); through `handleGamepadMenu`, a Start press pauses, holding Start does **not** re-toggle, and a second distinct press unpauses; A on gameover/title calls `startGame`; and A held during play does **not** confirm/restart (it's fire) — so A = fire and A = confirm never collide.
- **Binding-table data matches the F7 spec** — fire→A, shield→{RT,RB}, thrust→{D-Pad Up, LY−}, left→{D-Pad Left, LX−}, right→{D-Pad Right, LX+}, pause→Start, back→B, confirm→A; gameplay actions are `fixed:false` (rebindable in F8), menu/system actions `fixed:true`; keyboard defaults preserved.
- **End-to-end integration** — a gamepad D-Pad-Up thrust drives the **real** `Ship.update` through the unchanged `input.thrust()`: the ship accelerates and moves along +x at angle 0, confirming the input actually flows through, not just that the predicate returns true.

## Known issues

- **Audio unlock on a controller-only start is browser-dependent (v1.8).** A Start/A press makes a best-effort `AudioSys.init()`/`resume()` (same as any keydown), but the Web Audio autoplay policy generally requires a *user-activation* gesture, and gamepad input often doesn't qualify. So a player using only a controller may hear nothing until they press a key once. This is a known browser limitation, not a code bug — flagged as a playtest item. If it turns out to matter, the fallback is a one-time "press any key to enable sound" prompt (out of scope this phase).
- **Soft cap, by design (carried from v1.7):** the fire-block cap check is `count < maxBullets()`, so a Triple volley can transiently exceed the 12 cap by up to 2 bullets. Intentional (a clutter ceiling, not an exact quota). Flagged so it isn't mistaken for a bug.
- New watch items are in GDD §6: **controller feel & audio unlock** (v1.8) plus the still-standing **powerup drop rate & Magnet feel** (v1.7) and the v1.6 Hunter / v1.4 garbage-density / v1.3 knockback-chain items.
