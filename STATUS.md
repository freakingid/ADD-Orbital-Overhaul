# Asteroid Field Deluxe — STATUS

Last updated: 2026-07-06 · Build version: **1.9** (v2.0 Phase 8 of 9 shipped) · Last session: implemented **F8 — Pause Modal, Options Menu & Control Rebinding** (which also completes F7 by turning its hardcoded bindings into player-editable ones). Built the game's **first nested menu/UI state machine**: pause (`P` / gamepad Start) → `menu:root` (Continue / Options / Quit) → `menu:options` (SFX / Music / Master volume sliders + Controls + Back) → `menu:controls` (per-action keyboard+gamepad rebind grid + Return to Defaults + Back). `game.paused` is true **iff** a menu screen is open; `game.menu {screen,index,row,col,rebinding}` holds the cursor + capture state; both are toggled only by `openPause`/`closePause`/`quitToTitle`. Every screen is navigable by **keyboard or gamepad** through one `menuInput(action)` dispatcher (keyboard via the `keydown` handler; gamepad via `handleGamepadMenu` with edge-detected D-Pad/stick nav). **AudioSys was refactored to a gain-routing graph** — every voice now connects to an `sfx` bus → `master` bus → speakers, so the three sliders (`AudioSys.setVol`) attenuate real output; a `music` gain node exists but **nothing is connected to it (there is no music track — see below).** **Live rebinding** captures the next key/button into `bindings[action]`; **Return to Defaults** restores the five rebindable actions from `DEFAULT_BINDINGS` (a pristine snapshot taken *before* saved settings load). Volumes + custom bindings **persist to `localStorage`** (`afd_settings_v1`), all access `typeof`-guarded + try/caught so a storage failure never crashes. The critical correctness property — **menu/capture input never leaking into gameplay** — is enforced by resolving three input contexts in priority order (rebind-capture → menu-open → normal-play) *before* `keys{}` is ever written, plus `update()` staying frozen while paused.

## Working / verified

**Headless-verified this session (v1.9, `scratchpad/test-f8.js`, 40 assertions):**
- **State machine** — playing → pause opens `root`; down → Options; confirm → `options`; confirm Controls → `controls`; `back` walks options → root → (resume); the transitions land on the right screens/cursors.
- **Volume sliders drive real output** — left/right on a slider changes `AudioSys.vol[cat]` by `VOL_STEP` (0.1) **and** the routed gain node's `.gain.value` (SFX + Master both checked), clamped to `[0,1]` (SFX floors at 0, Master ceils at 1 — no gain > unity).
- **Live keyboard rebind via the REAL `keydown` path** — confirming a keyboard cell arms capture; the next key rewrites `bindings[left].keys` to `["j"]`, clears capture, and — crucially — is **NOT written into `keys{}`** (no gameplay leak). `input.left()` then fires on `j` live and no longer on the old default `ArrowLeft`.
- **Input separation / frozen-while-paused** — with a menu open, `ArrowDown` navigates the menu and is **not** recorded into `keys{}`; and `update(1/60)` while paused does **not** rotate the ship even with a gameplay key genuinely held. Contrast: unpaused, `ArrowLeft` *is* recorded and drives the ship.
- **Return to Defaults** — after mutating a keyboard binding (`left`→`j`) and a gamepad binding (`thrust`→button Y, axis dropped), `returnToDefaults()` restores **every** rebindable action to `DEFAULT_BINDINGS` exactly, including thrust's D-Pad-Up button **and** its analog `LY` axis.
- **Gamepad menu nav + pad rebind** — D-Pad Down moves the cursor once (holding it does **not** repeat — edge-detected via the `menuNav` latch); A confirms into Options; arming a gamepad cell then pressing Y captures Y as the binding and drops that action's analog axis.
- **`localStorage` persistence round-trip** (with an in-memory `localStorage` stub) — `saveSettings()` writes a structured blob; after corrupting the live state, `loadSettings()` restores the saved SFX volume (0.4) and the custom `fire` binding (`["k"]`).

**Regression re-run green this session:** `test-f2.js` (54/54), `test-f3.js` (28/28), `test-f4.js` (30/30), `test-f5.js` (47/47), `test-f6.js` (39/39), `test-f7.js` (52/52) — all pass unchanged (f7 drives the rewritten `handleGamepadMenu` heavily; the Start-pause / A-confirm behavior it asserts is byte-identical). `node --check` clean. `draw()` was runtime-exercised on all three menu screens + the capturing state (no throw).

**Not yet verified (needs a real browser — see playtest asks):** whether menu navigation *feels* natural on keyboard and on gamepad; whether a rebound control takes effect immediately in live play; and **whether settings actually persist across a real reload** (`localStorage` is stubbed headlessly and absent in the artifact sandbox — this is the first feature that persists anything).

## Known issues

- **Real-`localStorage` persistence is unverified (v1.9).** The save/load *logic* round-trips headlessly against a stub, but real cross-session persistence can only be confirmed in a browser running the file locally (the claude.ai artifact sandbox has no `localStorage`). All access is `typeof`-guarded + try/caught, so the worst case in the sandbox is "settings just don't persist" — never a crash. **Flagged as the top new playtest item.**
- **A gamepad rebind drops that action's analog-stick axis, by design (v1.9).** Capturing can only read a *button*, not a stick direction, and the on-screen binding must match what actually fires — so rebinding an action's gamepad control to a button removes its Left-Stick axis (e.g. rebinding thrust to a face button loses stick-thrust for that action). D-Pad still works if bound. **Return to Defaults restores the axis.** Called out so it isn't mistaken for a bug; revisit if it proves surprising in play.
- **Audio unlock on a controller-only start is browser-dependent (carried from v1.8).** A Start/A press makes a best-effort `AudioSys.init()`/`resume()`, but the Web Audio autoplay policy generally wants a *keyboard/pointer* user-activation gesture, and gamepad input often doesn't qualify — a controller-only player may hear nothing until one keyboard press. More relevant now that the whole pause/options menu can be driven by controller: if it's annoying, the fallback is a one-time "press a key to enable sound" prompt (still out of scope). Known browser limitation, not a code bug.
- **Soft weapon cap, by design (carried from v1.7):** a Triple volley can transiently exceed the 12-bullet cap by up to 2. Intentional clutter ceiling, flagged so it isn't mistaken for a bug.
- New watch item in GDD §6: **menu feel & first-time persistence** (v1.9), plus the still-standing **controller feel & audio unlock** (v1.8), **powerup drop rate & Magnet feel** (v1.7), and the v1.6 Hunter / v1.4 garbage-density / v1.3 knockback-chain items.

## Balance notes

- **Menu/options (v1.9) — no meaningful balance knobs.** `VOL_STEP` (0.1) is the per-nudge slider increment; volumes default to **1.0 (unity = pre-v1.9 loudness)**, so with no settings saved the game sounds exactly as it did in v1.8. The two-stage gain (sfx × master, both ≤ 1) can only *attenuate*, never boost past the original per-sound gains — no clipping risk introduced. Everything else in the menu layer is structural, not balance.
- **Carried from v1.8 (unchanged):** `GP_DEADZONE` (0.25) is still the one controller-feel knob (and it now also gates menu stick-nav).
- **Carried from v1.7 (unchanged):** `POWERUP_DROP_CHANCE` (0.10), `POWERUP_HEALTH_GAP` ([18,26]), `POWERUP_DURATION` (15), the weapon caps (Rapid 8 / Triple 12, combined = higher-of), `MAGNET_PULL`/`MAGNET_DAMP` (terminal ≈130 px/s), `ENGINE_MASS_MULT` (0.5).
- **Carried from v1.6 (unchanged):** the Hunter Satellite tables, `HUNTER_FLOOR_FRAC` (0.58), and the shared `RAMP_WAVES` (8).
- **Carried from v1.4 (unchanged):** `GARBAGE_DECAY` 12 s / `GARBAGE_SEVER_DECAY` 10 s, wave spawn `min(3+wave, 9)`, the mass-sum chain physics. **Still open: the joint F3+F10 Debris-density retune** — a post-playtest pass; with every combat/utility/UI system now in, this is the last big early-game balance lever left before the field feels "full."

## Playtest asks (Paul — can't be checked headlessly)

**TOP ASKS this session — all require a real browser (and a controller for the gamepad ones):** the menu *logic* is verified (40 assertions); what needs a human:

1. **Does menu navigation feel natural — on keyboard AND on gamepad?** Pause mid-game and move through Continue/Options/Controls with each.
   - **Keyboard:** arrows (and WASD) move, Enter selects, Esc backs out one screen, P resumes from anywhere. Do the volume sliders feel responsive at `VOL_STEP` 0.1 (10 steps 0→100%), or do you want finer/coarser?
   - **Gamepad:** D-Pad / Left Stick move (edge-detected — one press = one move; is that the right cadence, or do you want held-direction auto-repeat?), A selects, B backs, Start resumes. Does the whole menu feel reachable without touching the keyboard?
2. **Does rebinding a control actually take effect immediately in gameplay?** In Options → Controls, rebind (say) thrust to a different key and to a different gamepad button, resume, and fly. Does the new binding drive the ship right away (no reload)? Try Return to Defaults and confirm it snaps back. Two specific things to feel out:
   - **The gamepad-rebind-drops-the-axis behavior** (Known issues): after rebinding an action's gamepad control to a button, is losing that action's Left-Stick direction surprising, or fine? Return to Defaults restores it.
   - **Rebinding to a key the game already uses** (e.g. binding thrust to `P`) is allowed — no conflict detection. Does that footgun matter enough to warrant a warning, or leave it as "player's choice"?
3. **Do settings actually persist across a reload?** Change volumes and rebind a control, close the tab, reopen the file. Are your settings still there? (This is the **first** feature that persists anything, and it can only be confirmed outside the artifact sandbox — the headless test only exercises the serialize/deserialize logic against a stub.)
4. **Does a controller-only player get sound in the menu?** (Carried from v1.8.) If you navigate the whole pause/options menu by controller without ever touching the keyboard, do the menu blips / game audio play? If it's silent, that's the known Web Audio autoplay limitation — tell me if it's annoying enough to warrant a "press a key to enable sound" prompt.
