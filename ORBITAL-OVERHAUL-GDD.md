## 1. Vision & Design Pillars

A faithful-in-spirit knockoff of Atari's *Asteroids Deluxe* (1981), the harder sequel to *Asteroids*. Everything added to this game should be tested against these pillars:

1. **Vector purity.** All game objects are glowing stroked polygons on near-black. No fills (except tiny bullet dots and particles), no sprites, no textures. The blue-white palette echoes the Deluxe cabinet's blue-tinted mirror overlay.
2. **Momentum is the game.** The ship drifts. Mastery = managing inertia, not twitch aiming. Any new mechanic should interact with momentum, not bypass it.
3. **Pressure, not chaos.** Deluxe was famous for being *mean* — homing enemies, faster rocks — but every threat is readable and has a counterplay. New enemies need a visible tell and a fair answer.
4. **The shield is a resource, not a panic button.** Energy management is the core Deluxe innovation. New mechanics can spend, restore, or threaten shield energy.
5. **Greed is a choice.** (v1.1) Bonus systems like the salvage chain must always be optional, must degrade the player's core capabilities while pursued, and must be losable right up to the payoff. Reward the player for *choosing* risk, never force the risk on them.
6. **Ease players in.** (v1.5) Opening waves should be approachable; intensity ramps gradually with wave number, not all at once. Any threat parameter (speed, fire rate, accuracy, spawn frequency, density) is expressed as a value that interpolates from an easy wave-1 **floor** up to a full-intensity **ceiling** through one shared, wave-driven `difficultyFactor(wave)` curve (§2.13), so the whole game's pacing tunes from one knob (`RAMP_WAVES`). Complements Pillar 3 — pressure should arrive *on a schedule*, not slam the player at wave 1. As of v1.6 both the saucers (§2.6) and the Hunter Satellites (§2.5) scale off this curve.

## 2. Current Mechanics Specification

### 2.1 Ship
- Rotation 4.2 rad/s; thrust 340 px/s²; max speed 520 px/s; drag 35%/s (exponential).
- Collision radius 13 px. Wraps at the world edge.
- **Health:** a single HP pool (`SHIP_MAX_HP` = 250) for the whole run — no lives, no respawns. Unshielded hazard/bullet hits subtract source-specific damage; 0 HP is a permanent game over. See §2.7 and §2.12.
- **Hit-stun invulnerability:** any non-lethal hit grants `HIT_STUN_DURATION` (1.0 s) of invulnerability (rendered as blinking) during which further hits are ignored, and shoves the ship away from the hazard at `KNOCKBACK_SPEED` (250 px/s). There is no longer a separate 2.5 s spawn invulnerability — a run starts immediately vulnerable (the first wave spawns ≥220 px away). `ship.invuln` is now purely the hit-stun timer.

### 2.2 Weapons
- Max 4 player bullets alive at once (`MAX_BULLETS`); base fire cooldown `FIRE_COOLDOWN` 0.20 s; bullet speed 620 px/s + ship velocity inherited; 1.05 s lifetime. Bullets wrap.
- **The bullet cap flexes with the weapon powerups (v1.7, §2.14).** `maxBullets()` returns the base 4, or 8 with Rapid Fire, or 12 with Triple Shot (Triple also fires a 3-bullet spread); with both active it returns the higher cap (12), never the product. The fire block reads `maxBullets()` instead of the raw `MAX_BULLETS` constant.
- **Rapid Fire genuinely speeds up cadence (v3.0 P2, B-6).** The fire block sets its post-shot cooldown to `RAPID_FIRE_COOLDOWN` (0.09 s) when Rapid is active, else the base `FIRE_COOLDOWN` (0.20 s) — previously Rapid only raised the bullet cap, leaving cadence unchanged, which read as a bug. Triple Shot stays on the base cadence regardless of Rapid; Rapid+Triple together is therefore the strongest combo (fires fast *and* wide) — flagged for playtest, not rebalanced yet (FLAG B-6-a).

### 2.3 Shield (hold Shift)
- Energy 0→1. Drains 0.55/s while held; recharges 0.12/s while off; each deflection costs an extra 0.22. Cuts out below 0.02 energy.
- Radius 26 px. Deflects debris satellites and the large Hunter core elastically (bounced away at 1.1× their speed, pushed out of overlap). Blocks hostile bullets (costs deflection energy). **Destroys homing Hunters (medium/small) on contact** — costs a deflection's energy, and they still split (a medium shield-kill spawns 3 smalls that scatter outward), the same tension the old wedges carried (§2.5).

### 2.4 Debris Satellites (v1.4)

The passive main hazard (formerly "Asteroids" — the `Asteroid` class was renamed `DebrisSatellite` and its array `game.asteroids` → `game.debris` in Phase 3). They drift randomly and screen-wrap exactly as the old rocks did; the change is visual identity, split fan-out, and guaranteed salvage.

- **Three sizes.** Radius / base drift speed / score: large 46 px / 70 / 20 · medium 26 px / 110 / 50 · small 13 px / 160 / 100 (`DEBRIS_RADII` / `DEBRIS_SPEEDS` / `DEBRIS_SCORE` — scores unchanged from the old asteroid values).
- Speeds get ±30% random variance and a per-wave multiplier of `1 + (wave−1) × 0.08`.
- **3-way splits (F3).** Large → **3 mediums**; medium → **3 smalls**; small → destroyed (no children). This replaces the old 2-way split.
- **Guaranteed garbage at every tier (F3; count retuned v3.4 P1 — playtest knob).** *Every* destruction — including the final small-tier kill — emits exactly `DEBRIS_GARBAGE` (**4**, up from 3 in v3.4 P1) canisters, each with a random outward kick so they fan out. This replaces the old probabilistic single drop (`GARBAGE_DROP` by size). One fully-cleared large lineage (1 large → 3 mediums → 9 smalls = 13 kills) therefore yields **52** canisters, all mass 1.0 (see §2.10). The 3→4 bump compensates field-side for the same round's Hunter-drop tiering cut (§2.5) so total field supply holds roughly steady. This drove the Phase-3 balance pass (a since-removed decay timer, trimmed wave counts — §2.7, §2.10.1); as of v3.2 P3 garbage is permanent and density is held by coalescence + shatter instead of decay.
- **Shape (v3.3 P2 — authored satellite line art).** ⚠️ *Supersedes the procedural broken-hull silhouette.* v1.4 drew each debris as a procedural irregular hull polygon (`n = 6 + size` vertices) plus 1–2 antenna/panel shards; v3.3 P2 replaces that entirely with **`SAT_ART`**, a module-level table of **six authored, original stylized satellite silhouettes** — comms bus + twin solar panels, dish relay, boxy Earth-observer with a boom, a Sputnik-ish sphere with radial whip antennas, a cylindrical spent booster, and a truss/array segment. **Every one is wrecked** (a bent/snapped panel, a hanging antenna, a dented skin, a broken rail) — this is a debris field, not a working constellation. Each `SAT_ART` entry stores its polylines in **unit space** (all coords inside the radius-1 circle), each polyline tagged `closed`/open, and is drawn through the existing `drawPoly`/`glowStroke` in `COLOR.debris` (still the same blue, distinct from the teal Hunter satellites) — no fills, no new render primitive, no external assets (these are authored in code, not traced/imported — FLAG A-7, the file stays a self-contained GPL-3.0 HTML). Each instance picks one archetype at random (`this.art = SAT_ART[floor(rand(0, SAT_ART.length))]`; splits re-roll independently), bakes a per-instance copy **once** in the constructor — scaled to `this.radius` with a small random per-vertex jitter (~0.045·radius) for a "no two alike / wrecked" look, never re-randomized per frame — and `draw()` just iterates it, rotated by the existing random `this.angle`/`this.spin` (the spawn rotation, item 5, survives the draw-path rewrite; nothing anchors art to a fixed "up"). **Small-tier legibility gate:** at the small tier (`radius = 13`) a 10+ segment design is mush, so each archetype carries a **simplified `small` variant** (2–4 strokes: body + one feature) that the small tier draws instead of `full`. Collision stays circle-vs-circle at `this.radius` — the art is purely cosmetic and `DEBRIS_RADII` is unchanged. The title screen's decorative `titleDebris` uses `DebrisSatellite`, so it gets the new art for free.
- **Contact damage** carries over from F2 unchanged: `DEBRIS_DAMAGE` = large 50 / medium 35 / small 20 (`DMG_LARGE/MEDIUM/SMALL`).

### 2.5 Hunter Satellites (the Deluxe signature) (v1.6)

The active-homing threat — the *Asteroids Deluxe* signature enemy, reshaped. In Phase 5 the shipped killer `Satellite` + homing `Wedge` classes were **merged into one `HunterSatellite` class** with three size tiers (the Debris 3/2/1 convention: **3** large / **2** medium / **1** small). It reads as a **diamond** silhouette in teal (`COLOR.satellite`), staying visually and behaviourally distinct from the passive blue Debris Satellites (§2.4) — Debris drift, Hunters home — the readable tell Pillar 3 requires.

- **Last-stand pursuit (v3.6 P2b).** The passive-drift rule above holds only while Debris remain in the wave. Once `game.debris.length === 0 && !game.ship.dead`, every large core still on the field additionally steers its heading toward the ship every frame (wrap-aware `angleTo` + shortest-angular-delta turn) at `HUNTER_LAST_STAND_SPEED` (50 px/s) / `HUNTER_LAST_STAND_TURN` (0.5 rad/s) — both deliberately below even a wave-1 medium homer, so a wave can no longer stall forever on an un-huntable drifting core once it's the last satellite left. No new silhouette or tell marks the transition (FLAG-G: legibility here is a playtest question, not a shipped answer).
- **Large core (tier 3).** A symmetric diamond (radius 24) that **drifts passively** toward the ship's position *at spawn time* and never re-aims (confirmed design), tumbling slowly in place so it reads as "not aiming at you." Enters from just beyond a viewport edge relative to the ship. The **timed spawn** appears from wave 2 onward on a 14–22 s (first of a run) / 20–32 s (subsequent) timer, and that path is still **one lineage at a time** — it only fires when the whole Hunter family is clear (`game.hunterTimer`). A large core can *also* form by **garbage coalescence** (§2.5.1), and that path is **not** gated by the timer — coalesced cores are independent, so multiple concurrent lineages can exist (a coalesced core alongside a timed one, or several coalesced ones). Worth 200.
- **3-way splits at every splitting tier.** Shot large → **3 mediums**; shot medium → **3 smalls**; shot small → destroyed. (Replaces the old 3-then-2 fan-out.)
- **Medium (tier 2) & small (tier 1) actively home.** A forward-elongated kite-diamond (radius 15 / 10) that points along its pursuit heading. After a brief `HUNTER_SCATTER` (0.6 s) outward scatter, it relentlessly re-aims toward the ship every frame — wrap-aware, chasing across world edges. Worth 150 (medium) / 250 (small — the fastest, most agile thing in the game and the hardest to hit, so worth the most; the old small-wedge value preserved).
- **Difficulty-scaled from the start (F4 × F10).** Every tier's speed and turn rate is a *ceiling* the difficulty ramp (§2.13) climbs toward: each is `ramp(ceiling × HUNTER_FLOOR_FRAC, ceiling, wave)`, sampled once at spawn (wave is fixed for a Hunter's lifetime — Hunters block wave-clear). At wave 1 the whole family moves at `HUNTER_FLOOR_FRAC` = **0.58** of full speed (a gentle ~60%), ramping to ~96% by wave 20. Ceilings (`HUNTER_SPEED_CEIL` / `HUNTER_TURN_CEIL`): large drift **70 px/s**; medium **120 px/s / 1.6 rad/s**; small **175 px/s / 2.6 rad/s** (anchored on the old wedge numbers, extended across three tiers). The large core's passive drift *speed* scales too, but its behaviour stays passive at all difficulties.
- **Garbage scales by tier (v3.4 P1 — playtest knob).** `HUNTER_GARBAGE` is a per-tier table (`{ 3: 3, 2: 2, 1: 1 }`, shaped like `HUNTER_RADII`/`HUNTER_SCORE`/`HUNTER_DAMAGE`), replacing the old flat `HUNTER_GARBAGE = 3` (large/medium) + separate `HUNTER_SMALL_GARBAGE = 6` (small). A large kill emits **3** normal-mass (1.0) canisters, a medium kill **2**, a small kill **1** — still at `HUNTER_SMALL_MASS` = **0.5**, paler-mint-tinted (`COLOR.garbageLight`), the easy-to-haul bonus for finishing a Hunter line. A fully-cleared large lineage (1 + 3 + 9 = 13 kills) now yields `3 + 3·2 + 9·1` = **18 canisters** (9 normal-mass + 9 low-mass), total towable mass 9·1.0 + 9·0.5 = **13.5** — down from the pre-v3.4 flat-rate 66/39, compensated field-side by `DEBRIS_GARBAGE` 3→4 (§2.4) so total field supply lands close to the old level while Hunter drops scale by tier. **This is true of *every* Hunter regardless of origin** — the v3.2 "scrap-born Hunters drop nothing" exception is deleted (§2.5.1); single-garbage decay (§2.10.1) is the density governor, not emission suppression.
- **Contact damage (F2/F4):** `HUNTER_DAMAGE` = large **60** / medium **45** / small **30** — a step above the equivalent Debris tier (§2.12), since Hunters are the more dangerous, actively-seeking threat.
- **Shield interaction (§2.3).** The large core bounces off the shield elastically (you must shoot it); medium/small homers are **destroyed on shield contact and still split** (a shield-killed medium spawns 3 scattering smalls), preserving the old wedge tension — the shield answer and the "just shoot it" answer stay in tension, not stacked.
- **Chain vulnerability (carried forward).** A Hunter body touching a tow-chain node severs the chain exactly as the old satellite/wedge did (§2.10) — the Hunter survives; the touched node and everything aft break loose.
- **Kill path.** All of the above is handled by `destroyHunter(h)` (a flow function parallel to `destroyDebris`, not a `split()` method): score → boom → tier garbage → 3-way split (or, at the small tier, the low-mass burst and no children). Called both when a bullet hits and when a homer dies on the shield.

#### 2.5.1 Garbage coalescence → new Hunter (v3.1 P3 / v3.2 P1–P3)

Neglected salvage is a *standing threat*, not ephemeral clutter: uncollected canisters drift together into a **physical clump** — a heavy, growing body with real mass and size — that, once enough pieces have fused, collapses into a fresh large Hunter core. (As of v3.3 P4, a loose **single** canister ages out on `GARBAGE_DECAY` ≈ 22 s — §2.10.1 — but a **clump never decays**, so once pieces have fused they persist until the clump is shot, scooped, or transformed.) This is the "neglect breeds danger" pressure — the field you leave lying around can turn on you. Handled by `coalesceGarbage(dt)`, an O(n²) pass over free garbage run once per frame in `update()`, **after** the pickup/magnet loop and **before** `updateChain(dt)`. All measurement is wrap-aware (`dist2`/`shortDelta`).

> **v3.2 P1 SUPERSEDES FORK-2 (with a v3.3 P4 amendment).** The shipped v3.1 model kept a clump's mass fixed at 1.0 and let it tow as one normal canister — correct *only while clumps were hookable*. v3.2 P1 made a clump a real body — its **mass sums**, its **radius grows** (`7·√pieces`), its merges and attraction are **momentum-conserving** — and (P1) *un-hookable*, so the "clump tows as one mass-1.0 node" rule is gone. **v3.3 P4 (9c) then made clumps *scoopable* again** — but not as one mass-1.0 node: a scoop banks each piece as its own node at the clump's *per-piece* mass (the summed-mass model stands), lossily. So FORK-2's mass model is permanent; only its "one node" collection shortcut was ever wrong.

- **Activation delay (v3.3 P4 retune: 6.0 → 3.0 s — a playtest knob).** Every canister carries `coalesceDelay` (`GARBAGE_COALESCE_DELAY` = **3.0 s**) and a `pieces` count (default **1**), inherited by all three emission sites *and* `Garbage.fromNode`. A piece is **active** only once `coalesceDelay` has counted down to ≤ 0. **Both** the attraction and the merge require both pieces active, so a fresh burst rides its own explosion kick *apart* before it can start clumping. The 6.0 s value was tuned *for* permanent garbage; now that a single lives only ~22 s (`GARBAGE_DECAY`, §2.10.1), spending a third of its life inert meant pieces aged out before they could ever clump — hence the walk-back to ~3 s. `GARBAGE_DECAY` must stay a wide multiple of this delay or nothing clumps at all (the load-bearing relationship, asserted in test). Coalescence still grows from scrap that's been sitting and drifting, not from a just-shot cloud.
- **Mass, size, silhouette (v3.2 P1).** `pieces` stays the **1..11 threat counter** (at 12 it transforms). A clump's **`mass` is the sum** of its absorbed pieces' masses — a wad of 4 normal canisters is mass 4.0; a wad of 4 Hunter-scrap canisters is 2.0. Its **`radius` derives** as `7 × √pieces` (area-proportional; a single piece still reads the base 7, an 11-clump ~23 px — comparable to a large Hunter core). Both are recomputed at every merge.
- **Mass-weighted mutual magnetism (v3.2 P1; range retuned v3.3 P4).** Each active pair within `GARBAGE_MAGNET_RANGE` (**180 px** as of v3.3 P4 — was 260) attracts gravity-like: each piece accelerates toward the other in proportion to the *other's* mass — `a.v += (dx/d)·GARBAGE_MAGNET_PULL·b.mass·dt`, `b.v −= (dx/d)·GARBAGE_MAGNET_PULL·a.mass·dt`. This is momentum-conserving (`mₐ·accₐ == m_b·acc_b`), so heavy clumps become slow **anchors** that loose singles fall into. It **reduces exactly to the shipped symmetric pull when both masses are 1.0**, so `GARBAGE_MAGNET_PULL` needed no retune. The range was **raised** to 260 only because garbage was permanent (v3.2) and far-flung strangers had to find each other; with singles decaying again (v3.3 P4) a tighter 180 px range clumps a fresh burst without dragging the whole field into one super-clump — still **the number most likely to need tuning by feel** (FLAG D-1). An optional off-by-default speed clamp (`GARBAGE_CLUMP_MAXSPD` = `Infinity`) is a playtest knob against runaway clump speed.
- **Momentum-conserving merge (v3.2 P1).** Two active pieces within `GARBAGE_MERGE_DIST` (**12 px**) fuse: the survivor's velocity becomes the **momentum sum** computed with the *pre-merge* masses — `v = (mₐvₐ + m_b v_b) / (mₐ + m_b)` — its `mass` sums, `pieces += other.pieces`, `radius` is recomputed, and the other is marked dead. **This replaces the old literal vector sum** (`survivor.vx += other.vx`): with mass now summing and garbage soon permanent, a literal sum made every merge *add* speed, so clumps became the fastest things on screen — backwards from "a heavy wad of junk." Now a heavy clump barely speeds up when it eats a fast light piece. Low-mass Hunter scrap (mass 0.5) still counts as **1 piece** — the count is canister-based, the mass is mass-based.
- **Scoopable (v3.3 P4 9c — reverses v3.2 P1's "un-hookable" rule, FORK-6: unconditional).** A clump (`pieces > 1`) that enters the capture region (the base `GARBAGE_PICKUP` circle **or** the Scoop box, §2.10/§2.14 — with **no** `scoopLevel` requirement) while there's chain room is **scooped**: `room = cargoMax − chain.length`, `take = min(room, pieces)`, and `take` chain nodes are pushed at the clump's position, each at the clump's per-piece mass (`mass / pieces`). If the whole clump fit (`take === pieces`) it dies and its mass is conserved onto the chain; otherwise the leftover survives with `pieces`/`mass`/`radius` re-derived, its `coalesceDelay` re-armed, and an outward `SCOOP_SPILL_KICK` (≈ 85 px/s) kick directly away from the ship — "the rest of the ball floats off and away." Because a partial scoop can only happen *when the chain fills*, the leftover can't be immediately re-scooped (no room), so no cooldown is needed. **Scooping is LOSSY but instant:** greedily flying a 10-piece clump into a 3-slot chain banks 3 and throws 7 away. **The Magnet now pulls clumps too (v3.4 P4 — reverses the earlier "ignores clumps" rule):** with clumps directly scoopable, dragging one toward you is useful rather than noise, so the old `pieces === 1` gate on the pull is gone. The pull's `accel /= √mass` scaling keeps a heavy clump slow (a mass-11 clump feels ~30% of a single's pull — it drifts in but you can still outrun it), so the Magnet gathers clumps for the scoop without teleporting anchors into your lap. See §2.14.
- **Shatter (v3.2 P2) — the LOSSLESS clump tool (survives v3.3 P4).** One **player** bullet hitting a clump (`pieces > 1`, tested against its derived `radius`) destroys it via `shatterClump(g)` and emits exactly `pieces` fresh single canisters at its position — each with `mass = clump.mass / pieces` (per-piece mass identity is averaged back out; a deliberate fidelity trade to keep `mass` a single number, invisible in play), a **full, re-armed `coalesceDelay`** (the constructor default, so the wad disperses before it can start re-clumping — same knob as the activation delay above), and an outward `GARBAGE_SHATTER_KICK` (40–90 px/s) kick added to a fraction of the clump's own drift, mirroring the `destroyDebris` fan-out. A single canister (`pieces === 1`) is **not** a bullet target — player bullets still pass straight through it, exactly as before. **Hostile (saucer) bullets pass through all garbage**, clumps included — shattering is a player-only tool. **No score** is awarded for the shatter itself; the reward is the 2–11 now-hookable canisters. **Shatter and scoop are two doors onto the same clump, and the choice is the whole point (v3.3 P4):** shatter is **lossless** (every piece becomes a hookable single you can come back for) but takes a shot and leaves them loose to re-clump or decay; scoop is **lossy but instant** (fly through, bank what fits, waste the rest). Greed vs. tidiness — the same fulcrum v3.2 built, now with two tools on it. And a clump sitting at 9 pieces is still a countdown: harvest it (either way) for a near-full cargo run, or ignore it and let it hit 12 to fight what it becomes.
- **Cluster render (v3.2 P1) → single-hull silhouette (v3.3 P1) → scaled canister (v3.5 P2) → one safe green (v3.6 P1a).** ⚠️ *Superseded three times.* v3.2 P1 shipped a mini-canister cluster (golden-angle phyllotaxis offsets); playtest read it as "a pile of pickups," not "a target." v3.3 P1 replaced it with a 7–9 vertex jittered hull polygon (`makeClumpHull(radius)`, cached on `this.hull`, regenerated only at merge) specifically so a clump would read as **a target** — its rationale said so explicitly. **That rationale was written while clumps were un-hookable (v3.2 P1's rule).** v3.3 P4 (9c) reversed exactly that premise: a clump became directly *scoopable* — flying into one banks pieces onto the chain — so a hazard-shaped hull was now telling players to avoid the one thing they were meant to fly into. The art never caught up until now. **v3.5 P2 rewrites the render (and this rationale) to match:** a clump (`pieces > 1`) draws as ONE ordinary `drawCanister`, scaled by `this.radius / 7` (`= √pieces`, since `radius = 7·√pieces`) — a **larger canister**, not an asteroid-like hull. The scale factor makes the drawn long half-axis exactly equal `this.radius`, so the visual and the (unchanged) collision circle agree by construction, with no new tunable constant. `this.hull` and `makeClumpHull` are deleted entirely (constructor, the `coalesceGarbage` merge, and the partial-scoop leftover re-derivation all just drop the field). At the time, the canister color `lerpColor`'d from `COLOR.garbage` (`#c8ff50`) toward `COLOR.clumpHot` (`#ff5a2a`) as `pieces / HUNTER_COALESCE_COUNT → 1` — the only tell that a clump was nearing the Hunter transform. **v3.6 P1a deletes that red lerp and `COLOR.clumpHot` entirely.** A clump now picks colour by the exact same per-piece-mass rule as a single (`this.mass / this.pieces < 1 ? COLOR.garbageLight : COLOR.garbage`) — every garbage body on the field, single or clump, now glows one of two greens, never red. This is a deliberate trade, not an oversight: "is this safe to touch?" legibility was judged worth more than the free threat meter. **What's lost:** the clump's proximity-to-Hunter tell is gone; **size is now the only signal** (`radius = 7·√pieces` — a 1-piece single is 7 px, a 12-piece clump is 24 px — real and monotone, but much quieter than a color ramp). `lerpColor()` had no other caller and was deleted alongside `clumpHot`. The radioactive alpha flicker and `glowStroke` still come free from `drawCanister`, unchanged. `shatterClump`/`coalesceGarbage`'s merge math, the collision radius, and the scoop capture geometry are all untouched — only the render (and now the color rule) changed.
- **Non-solid to the ship (FORK-A2).** A clump does **zero** contact damage and is **not** added to the `hazards` array — the ship passes straight through. Garbage isn't the threat; what it *becomes* is. (The player is meant to shoot a clump apart, not bump it.)
- **Transform at 12.** When a survivor reaches `HUNTER_COALESCE_COUNT` (**12**) pieces it dies and a `new HunterSatellite(x, y, 3)` is pushed to `game.hunters`, with an `AudioSys.hunterborn()` warning cue (§2.8). This is **independent of `game.hunterTimer`** — coalescence can add lineages the timed spawn never would (multiple concurrent lineages, FORK-1). The existing end-of-frame dead-filter compacts the consumed pieces; no new filter.
- **Every Hunter drops garbage (v3.3 P4 9b — reverses v3.2 P3's FORK-B/B1).** *This bullet used to describe a `bornOfScrap` flag that suppressed garbage for coalescence-born lineages; that flag is **deleted** (FORK-5) and the suppression is gone.* Every Hunter of every origin — timer-spawned **or** coalesced from scrap — emits at its tier counts, so a full 13-member lineage drops `3 + 3·2 + 9·1 = 18` canisters (§2.10, §2.5's tiered `HUNTER_GARBAGE` as of v3.4 P1). **This still reopens the amplifier B1 had closed, just at a smaller multiplier than the original 9b landed at:** 12 pieces coalesce *in*, 18 come back *out* (~1.5×, down from the flat-rate era's ~5.5×/66 now that drops scale by tier). Under permanent garbage (v3.2) an unbounded amplifier would have diverged without bound, which is exactly why B1 existed. **What makes it safe is 9a:** single garbage decays again (§2.10.1), so the field is governed by *aging out* rather than by *emission suppression* — a much simpler, more legible lever, and one that doesn't require a hidden flag threaded through the split code; v3.4 P1's per-tier scale-down (§2.5) is a second, independent brake on the same amplifier. The coalescence transform still increments `game.stats.hunterCoalesced` once at the transform site — that stat **never read `bornOfScrap`**, so **Waste Not** (§2.17) is untouched: it still fires on a finished game with zero coalesced Hunters.
- **Coalescing vs recycling — the exits (v3.3 P4).** A loose **single** now has **three** exits: **recycled** (towed to the dock), **consumed into a Hunter** (coalescence), or **aged out** (`GARBAGE_DECAY`, §2.10.1). A **clump** has four: recycled, **scooped** (9c, banked directly), **shattered** back into singles, or **transformed** into a Hunter — clumps themselves never decay. Coalescing into a Hunter is the fate **Waste Not** watches (`hunterCoalesced === 0`). Shattering or partially scooping a clump is *not* itself an exit for the pieces that don't leave — they remain in the field as singles/leftover.
- **Magnet is the dominant anti-coalescence tool — and now it dominates the *whole visible field* (v3.4 P4).** The Magnet powerup (ship↔garbage pull, §2.14) is a separate, stronger, powerup-gated force. At its old ≈54 px reach it only vacuumed singles right next to the ship; at the new **`MAGNET_RANGE` = 380 px** it out-ranges `GARBAGE_MAGNET_RANGE` (the 180 px piece↔piece coalescence pull) *by more than 2×* and covers essentially the entire on-screen play area (viewport half-height is 360 px). **Consequence — on-design, but new in *degree*:** while a Magnet is up, nothing can clump anywhere the player can see, for the full 30 s duration — every loose single is being pulled to the ship faster than pieces can find each other. It still pulls clumps too now (scaled by √mass), so it also drags existing clumps in for the **scoop** (9c); a **shot** (shatter) remains the other way to reopen a clump. **⚠️ This screen-wide, 30-second coalescence lockout is this round's headline playtest ask** — whether that makes the Magnet feel appropriately powerful or simply erases the salvage-management layer whenever it drops.

### 2.6 Saucers

Shape, movement, scores, and bullets are the shipped v1.1 saucers; **as of v1.5 their pacing is wave-scaled** by the difficulty ramp (§2.13) so early waves are gentler and full intensity arrives around wave 20–25. Every scaled value interpolates `floor + (ceiling − floor) × difficultyFactor(wave)` from named floor/ceiling constants in the "Difficulty ramp & early-game pacing (F10)" block, so all early-game pacing tunes from the constants block. Behavior and shape are unchanged.

- One at a time. **Spawn gap** interpolates from a wide wave-1 floor `rand(20, 30) s` down toward `rand(12, 16) s` at full difficulty (re-expresses the old `rand(12,22) − min(6,wave)`; the first saucer of a run waits the wave-1 floor). Enter from a side edge, zig-zag vertically, exit the far side (no horizontal wrap; despawn after crossing).
- **Fire rate:** both saucers' between-shot intervals (and the first-shot delay) are multiplied by a **fire multiplier** that starts at `1.8×` at wave 1 (slower) and eases to `1.0×` at full difficulty, via the shared `Saucer.rollFireTimer(range)` method. Base ranges are `SAUCER_FIRE_BIG` (0.9–1.6 s), `SAUCER_FIRE_SMALL` (0.7–1.1 s), `SAUCER_FIRE_INIT` (0.6–1.4 s first shot).
- Big saucer: radius 20, 100 px/s, fires **randomly**. Worth 200. (Random fire is already the "easy" case, so only its fire *rate* scales — no accuracy change.)
- Small saucer: radius 12, 150 px/s, fires **aimed** — but the aim **error widens** from `±0.35 rad` (a barely-aimed spray) at wave 1 down to the shipped `±0.09 rad` precision at full difficulty. Worth 1000.
- **Small-saucer appearance chance** scales from `15%` at wave 1 to `60%` at full difficulty. This *folds in* the old score-gated logic (was: 25%, rising to 60% above 8000 points) — the escalation is now purely wave-driven so there's one difficulty system instead of two (§2.13, §2.10.1 rationale carries: greed is a choice, pressure on a schedule).
- Saucer bullets are red, 380 px/s, 1.4 s life, and destroy debris satellites (no score awarded, but they still emit their 3 canisters — see §2.4).

### 2.7 Waves, Health & Scoring

- Wave N spawns `min(3 + N, 9)` large Debris Satellites, placed in a ring 220–1100 px around the ship's *current* world position (see §2.11). *(Trimmed from `min(4 + N, 11)` in Phase 3 to offset the ~8× higher garbage volume from guaranteed 3-per-tier drops — see §2.10.1. As of v3.3 P4 single garbage decays again (the density governor), and this `min(3 + N, 9)` count is the direct lever for a fuller field if the pacing pass doesn't fully land it — see §9/the round's playtest asks.)*
- **Player-facing vocabulary vs. code term (v3.3 P1).** The HUD now reads **`LEVEL n`**, the player-facing word for what the code still calls `game.wave` throughout — `nextWave()`, `difficultyFactor(wave)`, `ramp(...)`, and every achievement keying off `game.wave` are all **unchanged**; this is a display-label rename only, so the two vocabularies (level = player, wave = code) don't drift apart. Previously the HUD showed `WAVE n` top-right at 22px in `COLOR.dim` — technically present but too dim/small/cornered to read; it's now `LEVEL n` at 22px in `COLOR.text`, directly under the score. **As of the CS009 HUD rebuild (P0–P6), Score and Level are the entire top-left readout** — Hull and Cargo moved into top-right rings and the powerup/Scoop rows into a bottom-left stack (§2.14 HUD layout), so nothing else shares this corner anymore.
- Next wave starts 2.5 s after the field is clear of debris, satellites, AND wedges — the exact gate is `game.debris.length === 0 && game.hunters.length === 0`. Free garbage and towed cargo do **not** block wave-clear; both persist into the next wave (the dock relocates).
- **Wave-clear gate** is `game.debris.length === 0 && game.hunters.length === 0` — unchanged, and still able to **rise** mid-wave: a Debris split (a large → 3 mediums, +2), a Hunter lineage arriving on its own 14–32 s timer (Hunters only appear from wave 2 on — they're absent at wave start), or, as of v3.1 P3, **a Hunter coalescing out of neglected garbage** (§2.5.1) — which, unlike the timer, can happen on *any* wave, at any time, and even off-screen (the `hunterborn` cue is the tell). A coalesced Hunter lands in `game.hunters` and **blocks wave-clear** until killed, same as any other. **The `TARGETS n` HUD readout that used to surface this count is gone (v3.6 P4)** — the left column was rebuilt around Score/Level/Hull/Shield/Cargo/Powerups and targets-remaining didn't make the cut; the gate itself, and its ability to rise mid-wave, is unaffected — only the display died.
- **Health, not lives (v1.3).** The ship has one HP pool (`SHIP_MAX_HP` = 250); there are no discrete lives, no respawns, and no continues — one hull for the whole run. Any unshielded hazard contact or hostile bullet subtracts source-specific damage, applies knockback, and opens a 1.0 s hit-stun window (see §2.12). At 0 HP the ship explodes, the tow load scatters, and the run ends (true game over).
- **The death is a spectacle — the `dying` state (v3.6 P5).** At 0 HP `killShip()` no longer freezes the field straight into `gameover`; it enters a new `dying` state for `DEATH_DURATION` (**2.5 s**) during which the world keeps simulating (via `updateDeath()`) — see §2.9 for the full state/control flow. Everything `killShip()` already did stays and is unchanged, **including the achievement flush** (`game.stats.gameEnded = true` + `Achievements.evaluate()` + `Achievements.save()`), which still fires exactly once, at `killShip`, so achievement timing is byte-identical to before. `killShip` additionally arms `game.deathT`/`game.shake`. During `dying`: an expanding shockwave ring (`DEATH_SHOCKWAVE_SPEED`, 700 px/s) is the detonation front — any Debris/Hunter it reaches detonates via the real `destroyDebris(a, false)` / `destroyHunter(h, false)` (**`awardScore = false` → no score and no achievement counters move; the chain-detonation is pure spectacle**), nearest-first falling out for free; the screen shakes (`DEATH_SHAKE_INIT`, decayed linearly); and a white flash washes the first frames (`DEATH_FLASH_TIME`/`DEATH_FLASH_ALPHA`). At `deathT ≤ 0` the state becomes `gameover` and the field freezes under the text as before. Music: `dying` isn't `title` or `playing`, so `musicStateFor()` falls through to `"off"` and the ~1 s `MUSIC_FADE_OUT` runs under the explosion.
- **Top-10 local high scores hang off the `dying → gameover` seam (v3.6 P6).** The instant `deathT` reaches 0 and `killShip`'s death spectacle hands off to `gameover`, `HighScores.qualifies(game.score)` decides whether the run earned a top-10 slot; if it does, `game.entry = { initials:[0,0,0], idx:0 }` arms a 3-slot arcade initials-entry UI on the gameover screen (§2.18) before the usual table/prompt appears. Non-qualifying runs skip straight to the (unhighlighted) table.
- **Score milestones repair the hull.** Every 10,000 points (`REPAIR_MILESTONE`) grants +25 HP (`REPAIR_AMOUNT`), capped at max; if already at full HP the milestone instead pays a flat `REPAIR_FULL_BONUS` (2,500) points so it's never wasted. This replaces the old "extra life every 10,000 points."
- **Health powerups also restore HP (v1.7, §2.14).** An ambient Health pickup restores +25 HP (`POWERUP_HEALTH_AMOUNT`, same value as the milestone repair), capped at max, on pickup — a second HP-restore path alongside the score milestone.

### 2.8 Audio (all synthesized, no assets)
- Fire: square-wave pitch drop. Explosions: filtered noise bursts, deeper/longer by size. Thrust: looped low-passed noise. Saucer: LFO-warbled triangle (higher/faster for small). Heartbeat: alternating two-tone sine thump whose interval shrinks with wave number and field density (floor 0.28 s). Shield ping / extra life: sine chirp. Menu blip (`ui`, v1.9): a short square pip, higher for confirm/select, lower for move/back/adjust.
- **`hunterborn` (v3.1 P3)** — the warning cue when a garbage clump coalesces into a Hunter (§2.5.1). Two low sawtooth voices (~196→138 Hz and ~139→98 Hz) a dissonant tritone (~√2) apart, both **descending** over ~0.4 s — deliberately dark, ominous, and distinct from `explosion`. Audio isn't spatialized, so it plays whether the Hunter forms on- or off-screen, making it the only tell for an off-screen coalescence.
- **Gain routing (v1.9, F8; +voice CS010 P9).** Every voice now connects through a two-stage bus — voice → **SFX gain** → **master gain** → speakers — so the Options volume sliders actually attenuate output (`AudioSys.setVol('sfx'|'master'|'music'|'voice', 0..1)`; 1 = unity = the pre-v1.9 loudness). A **music gain** node also hangs off master and is driven by the Music slider; **as of v3.4 P6 the music system (below) feeds into it** — the slider now governs real audio. **CS010 P9 added a fourth bus, `voice` → master**, driven by a new Voice Volume slider (§2.16) and fed by `VoiceSys` (below). `setVol`'s node picker is now a `this[cat]` lookup (not the old `master?…:sfx?…:music` ternary, which would have silently written `setVol('voice',…)` to the music gain); `loadSettings` derives the persisted-volume category list from `Object.keys(AudioSys.vol)` so a newly-added bus persists without a fifth edit site. All four volumes live in the frozen `afd_settings_v1` (additive — no schema bump).
- **Music (`MusicSys`, v3.4 P6).** A separate module **alongside** `AudioSys` (not inside it — `AudioSys` is a flat bag of one-shot voices; music needs a per-frame scheduler and a track graph). All music synthesis routes into `AudioSys.music`, so the existing Music Volume slider governs the whole system with no new plumbing. Signal path: **note envelopes → per-state `trackGain` (crossfaded) → `MusicSys.duck` → `AudioSys.music`**.
  - **Lookahead scheduler.** `MusicSys.update()` runs once per frame from the main loop. Each frame it schedules any note whose start time falls within the next `MUSIC_LOOKAHEAD` (0.2 s) window, using `ctx.currentTime` — a standard Web Audio lookahead. **No `setTimeout`/`setInterval` for note timing**, ever. Tracks are **DATA** (a generic `{ stepDur, steps, layers:[{type,gain,steps:[cell|null]}] }` step-sequencer table), never code, so v3.4 P7 adds its three gameplay tracks as more tables without touching the scheduler. `scheduleStep` clamps a note's start to `max(stepTime, currentTime)` so a stalled frame can never schedule into the past.
  - **State.** `MusicSys.setState(s)` crossfades between tracks: the previous `trackGain` fades to silence (and stops taking new notes) while a fresh one ramps in. Driven each frame off `game.state` **and the open menu screen** via `musicStateFor()`: **the High Scores screen wins first** — `(game.paused && game.menu.screen === "highscores") → "highscore"` (**CS010 P7** — the High Scores screen is a *menu*, so `game.paused` is true while `game.state` is still `title`/`gameover`/`playing`; `game.state` alone can't name it, so `musicStateFor()` also consults the active menu screen, reading `game.menu` in the same idiom it already reads `settings`); otherwise `title → "title"`, `playing → settings.musicTrack` (**v3.4 P7** — the selected gameplay track name IS the state string, so a mid-game track switch is just a normal `setState()` crossfade with no separate code path), else `"off"`. Switching in and out of the High Scores cue is likewise a normal `setState()` crossfade — no new track-switch path. `"off"` (game over) fades to silence over ~1 s (`MUSIC_FADE_OUT`) — a deliberate look-call, since the game-over screen is a quiet beat. **`gameover` deliberately stays `"off"`** even though the player may have just earned a high score there — the death spectacle ends in silence by design (§7 CS010 P7 records the open question of whether the cue should extend to the post-entry gameover table).
  - **The five shipping tracks + the frozen intensity machinery (v3.5).** The `tense`/`retro`/`ambient` set of v3.4 P7 was retired; the shipping music is now five synthesized step-sequencer tracks composed and auditioned in `tools/music-lab.html` and ported **verbatim** into the build (no re-tuning of a single gain — the lab is the source of truth for the note data): the title track **`title` (Beacon)** — Drift's engine turned hopeful, C major, the melody landing a Lydian ♯4 (F♯5) over a C pedal as the "departure" lift, no drums, ~21.8 s; and the four selectable gameplay tracks, ordered **calm → hot**: **`zen`** (the old title track's A-aeolian arc at 0.25 s resolution + subdued percussion, 48 s — the shipped **default**), **`derelict`** (a hum, two chromatic-mediant pads, and a distant ping; no beat, 24 s), **`drift`** (organ pulse, sub, pad, hymn lead; no percussion — **16 bars, A→B as of CS010 P6:** bars 0-7 are the original Am F C G Am F Dm E, bars 8-15 a brighter C-major turn (C G Am Em F C Dm E) with the arp an octave up and a fuller lead, resolving on E back into A; **43.5 s**), and **`warehouse`** (house — kick, melodic bass, 16th hat, two 7th-chord stabs, and — **as of CS010 P6 — a sixth `hook` layer**: **16 bars, an 8-bar Am7 Dm7 Fmaj7 G vamp played twice, A→B**, bars 0-7 groove only, bars 8-15 add a sawtooth-pluck riff on the offbeats. The hook was **cut from the round-2 track for being too present as a *constant* element** — the very reason it works as an intermittent chorus payload rather than an always-on line; **30.7 s**). All feed P6's scheduler unmodified, and they genuinely use the `noise`/`hp`/`drop`/`cutoffTo`/`q` layer fields whose `playNote` wiring landed in v3.5 P1. **Why the freeze, not a re-tier — two findings, both recorded so a future session doesn't repeat the work.** (1) **The real cause of "Retro has no melody":** `difficultyFactor(w) = 1 − e^-(w-1)/8` first crosses the tier-4 threshold (0.70) at **wave 11** — with the lead voice gated at tier 4, no gameplay track had an audible melody for the first ten waves, which is most of a typical run. That's the actual bug the old tiered `retro` was diagnosed for, not a synthesis problem. (2) **Re-tiering was tried anyway and rejected on audition:** the composed tiers 2-4 for the new five-track set were built, loaded, and played back at every intensity step — and on every track, the preferred mix was the **foundation layer alone**. The thickening read as clutter, not intensification. Difficulty in this game is already expressed through spawn rates, Hunter speed, and saucer cadence; it doesn't also need to be expressed in the arrangement. **THE FREEZE (the load-bearing decision, following directly from both findings):** no track carries a `tier` field, so every per-layer gate is built at `(layer.tier || 1) === 1` (always on) and `setIntensity()` skips every tier-1 gate — it walks the list and does nothing. The v3.4-P7 difficulty-gated intensity layering (Layers 2-4 unlocking on `difficultyFactor(game.wave)` past `MUSIC_LAYER_THRESHOLD`) therefore goes **dormant with no code change at all**: the machinery (`MUSIC_LAYER_THRESHOLD`, `MUSIC_LAYER_CROSSFADE`, `layerGates`, `setIntensity` and its `nextWave()` call site) is left fully in place — re-arming it later means adding `tier` fields back to track data and nothing else. **A layer's `tier`, if ever re-armed, must be in `1..4`** — `MUSIC_LAYER_THRESHOLD` has no key for 5+, and `f >= undefined` is always `false`, so a `tier >= 5` layer's gate would never open and the layer would be **permanently silent**. **Perf (FLAG-9c):** every layer is still always scheduled (gating is a downstream gain gate, and it's frozen on), so the worst case is intensity-independent — measured worst-case node creation for a single scheduled step across all five tracks is **13** (`zen` step 0, where bass+padLo+padMid+kick coincide), asserted `<= 16` in test, up from v3.4 P7's measured worst case of **7** (across the old `tense`/`retro`/`ambient`). That rise is not a regression: layer counts went from **4 tiers × 3 tracks** to **4-6 flat layers × 5 tracks**, so the system carries more tracks and more per-track layers for less worst-case cost per step than the tiered build it replaced. **CS010 P6 re-measured after the Drift/Warehouse B-sections (FLAG-6a): still 13** — the longer loops add no per-step cost (Drift's B keeps the same four-voice bar-start coincidence at 13; Warehouse's new `hook` sits only on otherwise-empty offbeat steps, so its worst step stays 11 and it is not the overall max). **CS010 P7 added a *sixth* track, `highscore` (Fanfare) — the celebration cue for the High Scores screen (below); its worst step is 12, so the overall worst case is still 13.** Track selection: `settings.musicTrack` ∈ `{zen, derelict, drift, warehouse}` (default `zen`), cycled in Options; `musicStateFor("playing")` returns it directly, so a mid-game switch is just a normal `setState()` crossfade.
    - **The High Scores cue (`highscore` = Fanfare, CS010 P7).** A sixth synthesized track, composed/auditioned in `tools/music-lab.html` and ported **verbatim** like the rest. It is **contextual, like `title`** — *not* a value of `settings.musicTrack`, so it is deliberately **absent from `MUSIC_TRACK_VALUES`** and never appears in the Options track picker; `settings.musicTrack`/`afd_settings_v1` are unchanged by this phase. Bright C major, an anthemic I–V–vi–IV lifted upward: a 16th bell arpeggio, a soft four-on-the-floor kick, a light shaker, and a triumphant triangle lead climbing to E6 at the peak and resolving back to C — 6 layers, 100 BPM, **8 bars, 19.2 s** (128 steps × 0.15 s). It plays **only while the High Scores screen is open** (`musicStateFor()` returns `"highscore"` there), and it is **exempt from menu ducking** (Ducking bullet below) — a track that plays *only* under an open menu would otherwise be permanently ducked. Same freeze constraints as every other track: no `tier` field, scheduler untouched, loop ≥ 10 s, no >1-step silence gap.
    - **Persistence consequence (expected, not a bug).** `loadSettings` accepts `musicTrack` only if it is in `MUSIC_TRACK_VALUES`, so a save written by an older build holding `"retro"`/`"tense"`/`"ambient"` (all removed) is now rejected and falls back to the default `"zen"`. No crash, no key rename — `afd_settings_v1` stays frozen.
  - **Ambient rebuilt (v3.5 P1), then superseded by Derelict (v3.5).** P1 rebuilt `ambient`'s tier-1 foundation from one note/8s to a continuously-present overlapping pad so selecting it no longer sounded like the music had stopped (§7 v3.5 P1 has the full density-measurement writeup). `ambient` itself is gone as of v3.5's five-track set, replaced by `derelict` — but the P1 finding behind it is still the reason `derelict`'s pads are voiced the way they are, so it's recorded here rather than left to rot in `ambient`'s deleted code: **the old foundation layer was a single sine oscillator at E2 (82 Hz) through a 300 Hz lowpass, and a pure sine that low has essentially no energy a laptop speaker can reproduce** — raising its gain to "match" `retro`'s square-wave bass could never have fixed that, because the problem was never level, it was **register and timbre**. `derelict`'s `hum`/`padRoot`/`padThird` layers are **detuned sawtooths**, not sines, and mostly voiced up around the root/third rather than dropping an octave below it, for exactly this reason — a sawtooth's harmonic series gives a small speaker something to reproduce, where a bare low sine gives it nothing.
  - **Ducking (§9 spec).** `MusicSys.setDuck(menuActive() && game.menu.screen !== "highscores")` each frame lowers `MusicSys.duck` to `MUSIC_DUCK_GAIN` (**0.5**) while a pause-menu screen is open (`menuActive()` ⇔ `game.paused`, which covers Pause / Options / Sound / Difficulty / Controls / Achievements) and back to 1.0 otherwise. **The High Scores screen is the one exemption (CS010 P7):** ducking exists to make room for gameplay audio *under* an overlaid menu, but the High Scores screen has its own full-level cue and nothing beneath it — without the exemption, a track that plays only while that menu is open would be permanently ducked to 50%. The move is a `linearRampToValueAtTime` over `MUSIC_DUCK_RAMP` (0.15 s) so it never clicks; `setDuck` is idempotent (only ramps on an actual change of target), so the extra clause costs nothing per frame.
  - **First-gesture gating.** Like every `AudioSys` voice, every `MusicSys` entry point early-returns when `AudioSys.ctx` is null, so nothing starts on page load — the title track begins on the first key/button press (and starts cleanly whenever the player returns to the title screen after a game, since `ctx` already exists). The `duck` node is built lazily (`ensureGraph`) the first time the graph is live.
  - **Four `playNote` voice fields (v3.5 P1, exercised for real by the v3.5 track set).** `playNote(layer, cell, t, layerIndex)` gained four optional per-layer fields, each an extension point on the existing envelope/filter chain, not a new signal path: **`noise`** — when truthy, the note's oscillator bank is replaced entirely by a shared noise-buffer source (`ensureNoiseBuf`) routed through the same envelope/filter sink, for percussion and texture that no oscillator can produce. **`hp`** — a highpass `BiquadFilterNode` at `layer.hp` Hz, inserted into the sink chain after the existing lowpass (`cutoff`/`cutoffTo`/`cutoffTime`/`q`) if present, thinning a layer's low end without a second gain stage. **`drop`/`dropTime`** — a per-note pitch-drop: each oscillator's frequency ramps from its starting pitch down by `drop` semitones (`exponentialRampToValueAtTime`, floored at 24 Hz so it can never ramp to/through 0) over `dropTime` seconds (default 0.06s), for a percussive "thump" attack on a pitched voice. **`q`/`cutoffTo`/`cutoffTime`** — `q` sets the lowpass filter's `Q` (resonance, default 1 if unset); `cutoffTo`/`cutoffTime` sweep the lowpass cutoff frequency across the note's own duration (a filter-envelope "wah"), independent of the amplitude envelope's `atk`/`rel`. All four are **inert unless a layer's data sets them** — the fields landed in v3.5 P1 with no shipped layer using them (a deliberately audible no-op ship, confirmed by test), and the v3.5 five-track set is what actually exercises every branch: `derelict`'s `hum`/pads use `cutoffTo` filter sweeps, `drift`'s `organ` layer uses `cutoffTo`+`cutoffTime`+`q` together, and `warehouse`'s drum layers use `noise` (hats) and `drop`/`dropTime` (kick).
  - **The title track (`title` = Beacon, v3.5).** One patient synthesized loop (~21.8 s = 128 steps × 0.17 s), **no percussion, no urgency**, deliberately its own mood — Drift's engine turned hopeful in C major, the melody lifting to a Lydian ♯4 over a C pedal as a sense of departure. Four WebAudio layers (a filter-swept square arpeggio, a low sawtooth sub, a detuned sawtooth pad, and a triangle lead), all synthesized — no audio files (house standard). It has no difficulty tiers (there is no difficulty on the title screen), which is now true of every track (the freeze, above).
- **`lowhp(on)` / `lowhpSet(t)` (v3.4 P5, rebuilt v3.5 P3)** — the low-health voice. Same persistent-voice shape as `thrust(on)`/`saucer(on)`: stored nodes created on `true`, torn down on `false`. **The original build** (v3.4 P5) was a gentle constant-amplitude 220 Hz sine LFO-swept ±18 Hz at 0.6 Hz, gain 0.05 — playtest/analysis found it the quietest voice on the whole SFX bus (peers: explosion 0.5, hit 0.16, fire 0.12, thrust/saucer continuous at 0.10/0.055) with no harmonics to cut through an explosion and, worse, **constant amplitude** — the ear adapts away a steady tone the longer it plays, so it got *less* noticeable over a long low-HP stretch; raising its gain alone would only have made it more annoying, not more noticeable. **v3.5 P3 rebuilt the voice's shape, not just its volume:** two static-level sine oscillators — a root at `LOWHP_ROOT_FREQ` (220 Hz, unchanged pitch) and a soft harmonic partner a fifth above (`LOWHP_HARMONIC_RATIO` 1.5, at `LOWHP_HARMONIC_GAIN_FRAC` 0.35 of the root's level, so it reads as an instrument-panel tone, not a test tone) — sum into a shared **amplitude-LFO "pulse" gain node** (an `lfo` oscillator → `lfoGain` → the pulse `GainNode`'s `.gain`, DC-offset by `LOWHP_PULSE_DEPTH`), turning the drone into a slow cockpit-warning **beep**: onsets get noticed, the gaps in between keep it bearable over a long stretch. Base gain moved into `thrust`'s neighborhood (`LOWHP_GAIN_MIN` 0.08 → `LOWHP_GAIN_MAX` 0.13, the pulsing does most of the perceptual work, so the range stays modest). **`lowhpSet(t)`** (new) scales urgency: `t = 1 − hp/LOW_HP_THRESHOLD` (0 at the threshold, ~1 near death) linearly interpolates the pulse LFO's rate (`LOWHP_PULSE_RATE_MIN` 0.9 Hz → `LOWHP_PULSE_RATE_MAX` 2.4 Hz) and the root/harmonic gain — gentle at the threshold, a real alarm near death. Called **every frame** the state is engaged (not edge-detected) from the same single site in `update()` (§2.12), right alongside `lowhp(true)`/`(false)`; it's a no-op if the voice isn't live or `ctx` is null, and idempotent on `t` so a steady-HP frame doesn't reschedule. Params move via `linearRampToValueAtTime` over `LOWHP_PARAM_RAMP` (0.15 s, same idiom as `MusicSys.setDuck`/`setIntensity`) — never a bare per-frame `.value` set, which would zipper. `lowhp(on)` itself is still driven from the single edge-detected call site in `update()` — never fired more than once per state transition — and torn down at every site `thrust(false)`/`saucer(false)` already are: `killShip()`, `quitToTitle()`, `openPause()`, plus the per-frame check when HP recovers. See §2.12 for the engagement rule.
- **Voice (`VoiceSys`, CS010 P9) — Dan speaks.** A Klatt-style formant speech synthesizer that gives the ship's engineer a small vocabulary of barked one-liners. Like `MusicSys` it is a **separate module alongside `AudioSys`, never inside it** (`AudioSys` is a flat bag of one-shot voices; speech needs a sequencer + a processing chain), and every entry point early-returns when `AudioSys.ctx` is null (headless-safe, first-gesture gated). Composed and auditioned in **`tools/voice-lab.html`** (CS010 §11a) and **ported verbatim** — same standing rule as `MusicSys`/music-lab: the acoustic engine (the ARPAbet `PH` table, `parsePhonTokens`/`buildUtterance`/`buildPitch`, and `VoiceSys._schedule`'s Web-Audio scheduler — `_render` split into `_emit`/`_schedule` in CS011 P2, see below) and the phrase data are byte-identical to the lab, tuned only in the lab. The lab's g2p text→ARPAbet authoring layer is **not** ported (its output — the per-line `phon` string — is already baked into `VOICE_LINES`; it is dead code in the game and stays in the lab, which is where a new line's `phon` is derived before pasting). **As of CS011, `VOICE_PARAMS` is no longer a single fixed object but a `let`-bound *active style*, re-pointed by `VoiceSys.setStyle(id)` at one of six entries in the `VOICE_STYLES` table (`comms`/`comms_f`/`flat`/`flat_f`/`vintage`/`vintage_f`; `comms` is the new default — the retired declarative voice is gone).** Those six styles and the ring-mod stage are ported **verbatim** from the second voice instrument, **`tools/voice-robot-lab.html`** (same port-verbatim rule; the lab's flanger/crush stages stay in the lab). `voiceEnabled()` (`settings.voiceStyle !== "off"`) is the enable gate; `VOICE_STYLE_VALUES` is the Off-first selectable-id list. **On-screen captions ship as of CS011 P2** (see the Captions bullet below); the menu picker and persistence are later CS011 phases (not yet built — `voiceStyle`/`captions` are runtime defaults only).
  - **Signal path.** A glottal pulse source (harmonic-tilt `PeriodicWave`) → a voicing envelope → a **parallel formant bank** (F1 direct; F2–F4 via a high-bus so nasal murmurs damp the highs) + a bypass frication path (Klatt parallel) → `VoiceSys.voiceBus` → a **ring-modulation stage (CS011)** (a sine `ringCarrier` drives a 0-baseline gain that multiplies the voice; `ringWet`/`ringDry` mix it per style, so a ring-*off* style is a transparent pass-through) → `ringOut` → the **dry path / radio-character chain** split (highpass/lowpass/tanh-shaper/compressor + a squelch-gated static bed) → **`AudioSys.voice`** (the volume bus) → master. **The dry/radio split now taps `ringOut`, not `voiceBus`** (the ring sits in front of both); the static bed still feeds the radio input **directly**, downstream of the ring. The lab's flanger and bitcrusher stages (`tools/voice-robot-lab.html`) are **not** ported — ring-mod is the only robot stage that ships. Formants ramp *through* per-consonant locus targets (the b/d/g cue); stops are closure→burst→VOT-delayed voicing; fricatives are spectrally typed.
  - **Lines are DATA.** `VOICE_LINES` is a table keyed by **event**, each event holding an **array** of alternatives (`{text, phon}`). Adding a new line for an existing event is a **one-line data edit**. Selection is a **plain random pick** each fire (CS010 §11d — not a shuffle bag). 29 lines across 9 `VOICE_LINES`-indexed trigger classes ship: hull-low / hull-relief / hull-full (3 each), the five powerup-collected and five powerup-expired barks, four dock-delivery tiers, "truck full," and — **as of CS011 P5** — five `chain_broken` frustration barks fired when the tow chain is knocked loose. **A 10th trigger class, level announcement (CS011 P4, below), is composed rather than picked** — it builds its `{text, phon}` inline in `VoiceSys.sayLevel(n)` from `numberToWords`/`NUM_PHON`/`LEVEL_PHON` instead of indexing `VOICE_LINES`, since the line's content is a function of `n`, not a fixed alternative.
  - **Cooldown & priority (§11e).** `VOICE_COOLDOWN` (1.2 s, playtest knob): no line starts while one is playing, or within that many seconds of the last one **ending** (the "never played" sentinel is `busyUntil = -Infinity`, so the first line — and any line in the AudioContext's first cooldown-window — is not swallowed). A per-event **priority** (`VOICE_PRIORITY`, default 1; 3 = hull critical, 2 = health milestones, level, and — **as of CS011 P5** — `chain_broken`) lets a higher-priority line **pre-empt** a lower one that is currently playing; an equal/lower one is **dropped**. **Superseded lines DROP — they never queue** (a queue would have Dan narrating events that finished ten seconds ago — worse than silence). **Final ladder: `health_low` 3; `health_relief`/`health_full`/`level`/`chain_broken` 2; everything else (collect_/expire_/dock_/cargo_full) 1 (default).**
  - **Captions (CS011 P2) — the visible echo.** `say(event)` is split into two methods sharing **one** cooldown/priority gate: **`_emit(line, p)`** resolves the gate once, then — for a line that PASSES — builds the utterance (`buildUtterance`, moved up out of the scheduler so a captions-only run needs no audio graph), shows the caption if `settings.captions`, and speaks it if `voiceEnabled()`; **`_schedule(utt)`** is the former `_render` body (the Web-Audio scheduler) taking that pre-built utterance. `say(event)` is now just the random line pick → `_emit(line, VOICE_PRIORITY[event]||1)`. Because caption and audio are the two outputs of the **same** gated `_emit`, a caption obeys the identical drop-not-queue/pre-empt rule as the audio — **captions are shown for exactly the lines that would have been spoken, and never late** — yet they are **independent of voice volume and of the Off style**: a player who selects voice Off (or mutes Voice Volume) still reads captions. `game.caption` (`{text, life, dur}`, on the `game` literal, reset in `startGame`, cleared in `quitToTitle`) is overwritten wholesale by **`showCaption(text, dur)`** (`life = 0.10 + dur + CAPTION_LINGER`) so a superseding line replaces the old, aged `-= dt` in `update()`'s playing body, and rendered by **`drawCaption()`** — a screen-space `drawText` centred at `VIEW_H − CAPTION_Y`, gated `game.state === "playing" && !game.paused && game.caption.life > 0` (so it freezes-out on menus/gameover, never sticking a stale line) and fading over its last `CAPTION_FADE` s via `ctx.globalAlpha`. `drawCaption()` is a **sibling** of `drawHUD()` in `draw()` (not inside it), so it is deliberately **independent of Capture's `H` HUD-hide toggle** — captions are accessibility text, not a HUD readout. Knobs (look-call): `CAPTION_LINGER` 1.5, `CAPTION_FADE` 0.4, `CAPTION_Y` 64, `CAPTION_SIZE` 20. (Level/chain caption lines are later CS011 phases.)
  - **Level announcement (CS011 P4).** Dan calls out the wave number at the start of every wave, including Level 1. `VoiceSys.sayLevel(n)` builds `{ text: "Level "+n, phon: levelPhon(n) }` and runs it through the standard `_emit` gate at `VOICE_PRIORITY.level` (2, same tier as the health milestones) — so it drops/pre-empts exactly like any other line, and the caption shows the numeral ("Level 23") while the audio speaks natural words ("Level twenty-three"), since the caption is built from `line.text` and the audio from `line.phon`. `levelPhon(n)` speaks `n` as words for `n <= 99` (`numberToWords(n)` → `NUM_PHON` lookups, joined on the engine's cosmetic `"/"` word-gap token) and falls back to digit-by-digit (`DIGIT_WORD` + `NUM_PHON`) for `n >= 100` — no attempt at "two hundred"-style composition past 99, since waves rarely reach three digits and the digit fallback is always intelligible. `LEVEL_PHON`/`NUM_PHON` are ported **verbatim** from `tools/voice-robot-lab.html`'s audited dictionary entries (same port-verbatim rule as the rest of `VoiceSys`), not re-composed in the build. Hooked in `nextWave()` immediately after `game.wave++` — since `startGame()` calls `nextWave()` for wave 1 right after `VoiceSys.reset()` clears the gate, Level 1 is announced on every fresh run, not just wave transitions.
  - **Triggers & latches (§11f).** Nine trigger classes, each hooked at one verified site (see §2.12/§2.14). **Hull-low** rides `update()`'s existing rising edge but gets its **own** latch, `game.lowHpVoiced`, that `openPause`/`quitToTitle` do **not** tear down (they reset the *siren* latch, which is why the siren rising edge re-fires on every unpause — the voice latch is what stops Dan re-announcing "Hull integrity is critical." each unpause; cleared only on the genuine falling edge + `startGame`). **Hull-relief** fires on the falling edge, additive to P3's `hpReliefFlash` exhale (parity, not a replacement). **Hull-full** is a new edge latch, `game.hpFullVoice`, on `hp >= SHIP_MAX_HP` — starts **true** at spawn (no "brand new ship" at wave start) and catches healing from **any** source (Health powerup and `addScore`'s repair milestone) because it lives in `update()`. **Powerup-collected** hooks `applyPowerup` (scoop in its early-return branch; Health stays silent — it speaks via the hull-full latch). **Powerup-expired** is the falling edge of **`powerActive(type)`** per timed type (routed through `powerActive`, never `powerFx`, so it catches count modes too). **Scoop-lost-a-level** hooks the damage-decay site in `damageShip`. **Dock delivery** (FORK-5 = b) fires on the pop that **empties** the chain, sizing the haul by `game.deliveryCount` (5–9/10–14/15–19/20+ tiers; 1–4 says nothing) — the emptying pop happens exactly once, so **the gate is the latch** (`breakChain`/`scatterChain` also zero `deliveryCount` but never reach the site, so they stay silent). **Truck-full** fires on the pickup where `chain.length` reaches the **runtime** `game.cargoMax` (12→24) — again gate-as-latch, re-arming after a delivery. **Chain-broken (CS011 P5)** fires from `breakChain(i)`'s single choke point — the ONE place a hostile bullet or a Hunter body knocks a chain node loose — so both call sites (a bullet hit, a Hunter collision) route through it and a multi-node break (several nodes severed on the same hit) still speaks once, collapsed by the same cooldown/priority gate every other line uses. `scatterChain()` (ship death scatters the whole load) is **deliberately NOT hooked** — the death spectacle (§2.7 `dying`) already carries its own weight; Dan does not editorialize over it.

### 2.9 States & Controls
- States: `title` → `playing` → **`dying`** → `gameover` (the `dying` death spectacle inserted between play and game-over in **v3.6 P5** — §2.7). Gameplay: rotate ←→/A/D, thrust ↑/W, fire Space, shield Shift — all **rebindable** as of v1.9 (§2.16).
- **A qualifying `gameover` may arm high-score initials entry (v3.6 P6 — §2.18).** While `game.entry` is set, both input paths (keydown's normal branch and the gamepad's edge-detected nav) route up/down/left/right/confirm to `entryInput()` instead of their usual title/gameover meaning, and **return before reaching the confirm→`startGame()` check** — belt-and-suspenders with that check also gaining an explicit `&& !game.entry` guard, since without either, a confirm that just committed the entry (clearing `game.entry`) would fall through and restart the run in the same keypress (the load-bearing bug this phase's spec flagged). `O`/`B` (system menu) and Start/Back are likewise inert while entry is live. Once committed (or immediately, for a non-qualifying run), the normal `gameover` controls (Confirm restarts, `O`/`B` open the system menu) apply as before.
- **The `dying` state is input-inert and unskippable (v3.6 P5).** For its `DEATH_DURATION` (2.5 s) the reduced sim (`updateDeath()`) takes **no ship input, runs no spawns/pickups/collisions**, and every `game.state` read site was audited so the spectacle can't be interrupted: **Confirm/Enter does not start a new game** (the `startGame()` site gates on `title`/`gameover` only — a player can't skip or restart the explosion), the pause key is inert (**FLAG-H:** no pausing mid-death, by design), and the `O`/`B` system-menu and gamepad `onTitleOrOver` paths are all excluded. The one intentional exception: **`Capture`'s `P` screenshot key IS honoured during `dying`** (`Capture.canShoot()`; FLAG-I — plausibly the single frame most worth capturing), while `O` slow-mo and `H` HUD-toggle stay live-play only. The `dying → gameover` handoff then re-enables the normal `gameover` controls (Confirm restarts, `O`/`B` open the system menu).
- **Corrected control scheme (v3.0 P4 — supersedes the earlier B-1-a mapping).** The menu system is reachable from *every* state, and the controls read the same on keyboard and controller:
  - **Start a game** (from `title`/`gameover`): **Enter** or controller **A**, *or* controller **Start**.
  - **Pause** (during `playing`): **ESC** or controller **Start**. (`P` is retired — ESC is the pause key.)
  - **Open the Options/Achievements system menu** (from `title`/`gameover`): **O** or controller **B**. Its root is **Options / Achievements / Back**; Back closes the overlay back to the underlying screen.
  - **Controller Start is a session toggle**: title/gameover → start a game; playing → open pause; while paused → dismiss & resume.
  - **ESC and controller B are context-aware**: ESC = pause outside a menu / back inside one; B = open the system menu on title/gameover / back when a menu is open. In-menu, **Enter/A** confirm, **ESC/B** back, **Start** resumes.
- **Gamepad (v1.8, §2.15).** The first connected controller works alongside the keyboard: D-Pad / Left Stick rotate + thrust, A fires, right trigger/bumper raises the shield. System reads per the corrected scheme above (Start = session toggle, A = confirm/start, B = open-menu/back). Gameplay defaults are player-editable via Options → Controls (v1.9, §2.16).
- **Pause menu / Options (v1.9, §2.16).** The root is context-aware: **Continue / Options / Quit** when paused mid-game, **Options / Achievements / Back** as the title/gameover system menu. Options has SFX/Music/Master volume sliders + a Controls rebinding screen + an **Achievements** viewer (v2.0, §2.17); navigable by keyboard **or** gamepad. Settings persist via `localStorage`.

### 2.10 Radioactive Salvage, Tow Chain & Recycling Dock (v1.1)

**Garbage drops (F3-updated; count retuned v3.4 P1).** Every destroyed Debris Satellite emits exactly `DEBRIS_GARBAGE` (**4**, up from 3) radioactive canisters at *every* tier, including the final small-tier kill (52 per fully-cleared large lineage — see §2.4). This replaced the old probabilistic single drop. Canisters inherit 25% of the debris's velocity plus a random outward kick (so the trio fans out), drift, spin, and screen-wrap. Each canister carries a **`mass`** field (F5; default **1.0** for all Debris-sourced scrap — see below). **A loose *single* canister decays again (v3.3 P4 9a).** `GARBAGE_DECAY` (≈ **22 s**) counts down in `Garbage.update()` **only while `pieces === 1`**; at 0 the canister dies, blinking out over its last `GARBAGE_FADE` (≈ 2 s) as the expiry tell. A **clump** (`pieces > 1`) never decays (FORK-4) — it's a target and a scoopable pickup, so aging it out would steal loot and the Hunter-prevention window. Severed garbage (`Garbage.fromNode`) is a single and inherits the same one `GARBAGE_DECAY` (the old separate `GARBAGE_SEVER_DECAY` stays deleted). So a single now leaves the field by **three doors** — **recycled**, **consumed into a Hunter** (coalescence, §2.5.1), or **aged out** — and decay is the density governor that lets *every* Hunter drop garbage again (9b) without the field diverging (§2.10.1).

**Cargo mass (F5).** Each `Garbage` and each chain node carries a `mass` (default 1.0). `Garbage.fromNode()` carries `mass` back when a node is severed to free garbage; pickup copies the canister's `mass` onto the new chain node. The chain-handling penalties (below) scale with the chain's **total mass** — `chainMass()` = Σ node.mass — not its node count. All current (Debris-sourced) scrap is mass 1.0, so a chain's mass sum equals its length and handling is unchanged from v1.3; lower-mass scrap (planned for Hunter Satellites, F4) will tow more easily. See §3.4.

**Collection & chain.** Flying within 18 px (`GARBAGE_PICKUP`) of a **single** canister (`pieces === 1`) hooks it onto a tow chain behind the ship (up to the current tow cap `game.cargoMax`; extras stay in the field). **A clump (`pieces > 1`) in the capture region is now *scooped* (v3.3 P4 9c — reverses v3.2 P1's "un-hookable" rule):** as many of its pieces as the chain can hold become nodes at the clump's per-piece mass, the overflow is discarded (a lossy but instant harvest — full mechanics in §2.5.1). It's **unconditional** — no `scoopLevel` requirement; the scoop box just makes it *easy*. The Magnet now pulls clumps toward the ship too (v3.4 P4, √mass-scaled — §2.14), gathering them for the scoop. **The capture region grew a directional mouth (v3.3 P3, §2.14 Scoop).** The base 18 px circle **always still applies**, but it is now **OR'd** with an oriented capture **box** on the ship's facing axis whose size scales with `game.scoopLevel` (0–5): via `inScoopBox(g)`, the wrap-aware ship→garbage delta (`shortDelta`) is projected into ship-local space and a piece is captured if `|lateral| ≤ SCOOP_WIDTH[lvl]/2` **and** `−SHIP_RADIUS ≤ forward ≤ SCOOP_DEPTH[lvl]`. At `scoopLevel 0` the box is empty (width/depth 0), so collection is **byte-identical to the pre-scoop build** (circle only). The Magnet's `MAGNET_PICKUP_MULT` multiplies the **circle** only, never the box, so the two never double-count. The cap is a **runtime value that grows during a run** (§2.10.2) from `CARGO_BASE` (12) toward `CARGO_CAP_MAX` (**24**, up from 20 in v3.4 P1). The chain is simulated: verlet nodes with 20 px distance constraints, anchored to a hitch point 12 px behind the ship, relaxed over `CHAIN_ITER` (4) passes/frame. Physics effects:
- *Mass penalty:* thrust is divided by `1 + m×CARGO_THRUST` and top speed by `1 + m×CARGO_MAXSPD`, where **`m` = the chain's total mass** (`chainMass()` = Σ node.mass, F5). Coefficient history: B-8 softened them (`CARGO_THRUST` 0.10→0.06, `CARGO_MAXSPD` 0.05→0.03) so a full chain at the higher cap matched the old full-12 feel; **CS010 P2 firmed them** (`CARGO_THRUST` 0.06→**0.07**, `CARGO_MAXSPD` 0.03→**0.035** — playtest knobs) because a mid haul wasn't reading as heavy. Now a *full chain* (m = 24) ≈ **37% thrust / 54% top speed** (was ~41%/58%), and a *base-12* chain ≈ **54% thrust / 70% top speed** (was ~58%/74%). Note the divisor `1/(1+m·k)` **asymptotes**, so raising `k` can't make the mid range bite hard without over-penalizing the full chain — which is why CS010 leaned on the **tug** (`CARGO_MASS`, §3.4) for mid-range heft and shipped `CARGO_TURN` (a turn-rate penalty, dormant at 0) as the real "inertia" fix. The "cap is a handling ceiling" rationale (§2.10.1) is preserved. (For all-mass-1.0 cargo, m equals the node count.)
- *Momentum tug:* when the first link is stretched taut, the ship receives a pull-back acceleration of `CHAIN_TUG (26) × stretch-px × massFactor` (massFactor = min(1.4, m×`CARGO_MASS`), same total-mass `m`). `CARGO_MASS` is **0.10** (CS010 P2, up from 0.07 — a PLAYTEST KNOB): the 1.4 tug cap is now reached at **m=14**, so a mid haul feels the momentum pull-back sooner — a base-12 chain tugs at **1.2** (was 0.84). **The 24-node ceiling still doesn't make the tug worse:** `min(1.4, m×0.10)` saturates at m=14, so a full 20-chain and a full 24-chain both land at the same capped 1.4 — the CS010 raise strengthens the **low-to-mid** tug without touching the full-chain endpoint, which is exactly why the tug was the lever chosen for "towed mass should be felt" (§2.10.2). Hard turns and burns get the ship yanked by its own cargo.
- The chain is wrap-aware and follows the ship across screen edges.

**Chain vulnerability.** Any debris satellite, wedge, or killer-satellite body — or any hostile bullet — touching a chain node destroys that canister; **everything aft of it breaks loose** as free garbage (scatter velocity, mass preserved). Severed scrap is a single, so (as of v3.3 P4) it inherits the one `GARBAGE_DECAY` life clock like any other single — knocked-loose canisters drift, coalesce, get re-collected, or age out. A chain node itself never decays (it lives in `game.chain`, not `game.garbage`). Ship death scatters the entire load. Player bullets pass through **single** canisters and the chain (they're debris, not targets) — but as of **v3.2 P2**, a player bullet hitting a free **clump** (`pieces > 1`) is a target: it shatters the clump into hookable singles (§2.5.1). Hostile bullets still pass through all garbage, clumped or not.

**Recycling dock.** A green octagonal station (radius 44) placed randomly each wave, 260–900 px from the ship's *current* world position (see §2.11). Non-solid — purely a delivery zone. While the ship is within radius+10 and carrying cargo, canisters peel off the **tail** every `DOCK_OFFLOAD_INTERVAL` (**0.05 s** — v3.4 P1, was a bare literal `0.13`). Delivery score escalates within a single visit: `50 + 25×(n−1)` for the nth canister (full 12-chain = 2,250). The combo counter resets when the ship leaves the dock's neighborhood (radius+40). Delivery pitch rises per canister; score popups float up. **Dock size is a difficulty lever (v3.4 P2, ships DISABLED).** `Dock.radius` is `DOCK_RADIUS * leverScale(LEVER_DOCK_SIZE, game.wave)`, baked once per `new Dock()` (a fresh dock every wave, so a re-size only ever happens at the wave boundary, never mid-wave). Every other read of the radius — the intake-vane geometry, the "RECYCLE" label offset, the offload-proximity gate (`+10`), and the combo-reset distance (`+40`) — routes through `dock.radius`/`game.dock.radius`, with the `+10`/`+40` feel margins kept as absolute px (not scaled). With the lever disabled, `LEVER_DOCK_SIZE.start` (2.0) is pinned at every wave, so the dock is simply **2× its old size** — see §2.13 and `DIFFICULTY-LEVERS.md` for the mechanism and the registry.

**HUD.** A CARGO ring (CS009 P3, superseding the earlier `CARGO n/{cargoMax}` left-column text readout) sits top-right beside the HULL ring (§2.12) — same construction: a dim full-circle track, a value arc at `game.chain.length / game.cargoMax` (the current runtime cap 12→24, read live every frame, never cached), and center text `n` over `/cargoMax`. **Full (`chain.length >= cargoMax`) or a fresh cap-up (`game.cargoFlash`, armed for `HUD_CAP_FLASH` 0.5s whenever `cargoMax` grows) both turn the ring gold (`COLOR.ach`) and pulse/thicken** — `COLOR.ach` means exactly one thing everywhere in the HUD, "this resource is at its cap; more is wasted," and this ring is one of the two places it appears (the other is the HULL ring's MAX state, §2.12). The center number's color is a separate tell (`COLOR.text` when non-empty, `COLOR.dim` when empty) and is never gilded, only the ring arc is. A green chevron orbits screen-center at radius 42, pointing toward the dock, whenever cargo > 0.

**Recycle-hub powerup emission (v3.6 P3).** The dock is now also one of the three powerup sources (§2.14): the first time a visit's delivered-canister count passes through **10** (one line above the existing `=== 12` Heavy Hauler check — same "exactly once per visit" idiom), the offload block emits one powerup from the dock's own position at `DOCK_POWERUP_SPEED` (120 px/s) in a random direction, plus a "SALVAGE BONUS" `FloatText` marking the cause.

#### 2.10.2 Tow-capacity progression (v3.0 P6 / B-8)

The tow cap is no longer the fixed `CHAIN_MAX = 12`. It's a runtime `game.cargoMax` that **starts at `CARGO_BASE` (12) and grows by +1 per `CARGO_GROW_PER` (30) canisters recycled (delivered) in the current run, bounded by `CARGO_CAP_MAX` (**24**, v3.4 P1, up from 20)**. Growth is checked at the dock-offload site right after the per-run delivered count increments, so the cap climbs at most one step per canister; each increase pushes a brief **"TOW +1"** float. `startGame` resets `cargoMax` to `CARGO_BASE`, so capacity is earned fresh each run.

- **Why per-delivery, not per-wave (FORK A-fork-2):** tying capacity to *engaging the salvage loop* directly rewards the greed mechanic (Pillar 5) and self-paces — heavier haulers earn heavier hauling — rather than passively handing capacity out by wave.
- **The physics retune is the load-bearing half.** A naive m=24 chain under the old (pre-B-8) coefficients would be far into the "broken, not heavy" zone §2.10.1 warns about. The softened coefficients (§2.10) already put the full-cap chain in a deliberately-heavier-but-still-controllable band; v3.4 P1 raised the cap 20→24 **without** re-softening them (m=24 then landed at ~41% thrust / ~58% top speed). **CS010 P2 firmed the coefficients** (thrust 0.06→0.07, top-speed 0.03→0.035, tug 0.07→0.10 — all playtest knobs) so a full 24-chain now sits at **~37% thrust / ~54% top speed** and a *mid* haul finally reads as heavy — a bigger hold is *supposed* to cost handling. Stability was re-validated at each step rather than assumed (`CHAIN_ITER` stays **4**; the 24-node hard-thrust-flip-across-wrap stress holds at worst-case link stretch ≈ **4.11 px** on the 20 px `CHAIN_LINK` under the documented methodology — unchanged by the CS010 retune, which is stability-neutral — see §3.4).
- **"Full chain" achievements decouple from the cap (FLAG B-8-b).** Heavy Hauler / Freight Baron / The Long Haul must key on a **fixed ≥12** delivered in one dock visit, *not* on `game.cargoMax` — otherwise their meaning drifts as the cap grows. The existing full-visit hook keys on a literal `12` (`game.deliveryCount === 12`), confirmed still decoupled from `cargoMax` as of v3.4 P1; the achievements-overhaul phases own keeping it fixed.

> Constants live in the salvage/chain block: `CARGO_BASE` (12), `CARGO_CAP_MAX` (**24**), `CARGO_GROW_PER` (30), the retuned `CARGO_THRUST`/`CARGO_MAXSPD`/`CARGO_MASS`, `CHAIN_ITER` (4), and `DOCK_OFFLOAD_INTERVAL` (**0.05**, v3.4 P1).

#### 2.10.1 Design considerations & rationale (read before changing this system)

These were deliberate calls, not accidents — future sessions should understand *why* before tuning:

- **Guaranteed per-tier drops (F3, count retuned v3.4 P1) supersede the old inverted probabilistic model.** The v1.1 design dropped smalls reliably and larges rarely (≈5 per lineage) so garbage appeared only where a lineage was *finished*. Phase 3 replaced that with a flat 3 canisters at every Debris tier; v3.4 P1 bumped it to **4** (52 per fully-cleared large lineage) to compensate for the same round's Hunter-side garbage cut (§2.5) — a deliberate, request-driven volume increase to make salvage a much bigger, more constant temptation (Pillar 5: greed is a choice). The player is **not** expected to collect it all — the point is a field rich with *optional* risk/reward, not housekeeping. As of v3.3 P4, a loose **single** decays again (`GARBAGE_DECAY` ≈ 22 s — see the next bullet) while **clumps are permanent** until shot/scooped/transformed; density is held by that single-decay governor *plus* the consuming forces below (coalescence, scoop, shatter-to-harvest) and the tow ceiling (`game.cargoMax`, 12→24 — §2.10.2).
- **Single-garbage decay is the density governor again — because *every* Hunter now drops garbage (v3.3 P4 9a+9b — supersedes the v3.2 "density is held without decay" rule).** *The history of this bullet is instructive, so read it before touching decay again. v1–v3.1: a 12 s `GARBAGE_DECAY` timer was the primary density lever; the blink-out was the player's countdown. v3.2 P3 **removed decay entirely** and argued permanent garbage was fine because coalescence (12 pieces in → 1 Hunter) consumed pieces and a `bornOfScrap` flag made coalesced Hunters drop **0** garbage back out, closing the loop at 12-in/0-out. v3.3 P4 reverses BOTH of those:* every Hunter drops garbage again (9b deletes `bornOfScrap`), which **reopens the 12-in / up-to-N-out amplifier** that suppression had closed (N was 66 under the flat-rate `HUNTER_GARBAGE`/`HUNTER_SMALL_GARBAGE`; v3.4 P1's per-tier `HUNTER_GARBAGE` table brought a full lineage's drop down to **18**, so the amplifier is now ~1.5× rather than ~5.5×, a second brake on top of decay). **Why reopening it is correct, not a regression:** the `bornOfScrap` flag was a hidden piece of state threaded through the whole split lineage purely to plug a divergence — legible only to someone who'd read this section, and it forced the odd rule that a scrap-born Hunter drops nothing. Re-introducing decay replaces that special-case plumbing with a single, visible, universal governor: a loose **single** ages out on `GARBAGE_DECAY` (≈ 22 s), so no matter how much a Hunter drops, uncollected scrap self-limits. Decay handles the *floor* of clutter; coalescence, scoop, and shatter-to-harvest handle the *ceiling* (they give the player reasons to engage the pile). **Clumps do not decay** (FORK-4) — only singles — so the coalescence/scoop/shatter loop is never robbed of its target. **The load-bearing relationship:** `GARBAGE_DECAY` must stay a wide multiple of `GARBAGE_COALESCE_DELAY` (≈ 22 vs ≈ 3) or singles age out before they can ever clump — asserted in test, not just documented. If a future session finds late-wave clutter, the levers are `GARBAGE_DECAY` (shorter → field clears faster), `GARBAGE_MAGNET_RANGE` (180, most likely to be wrong — FLAG D-1), and `GARBAGE_COALESCE_DELAY`; if the field feels *empty*, lengthen decay or the `nextWave` count (§9 lever). **Do not re-remove decay** to "simplify" — 9b's universal emission depends on it.
- **The tow cap is a handling ceiling, not a scoring one.** Beyond a node count the thrust penalty makes the ship feel broken rather than heavy, and long constraint chains start whipping. The cap now *grows* to `CARGO_CAP_MAX` (**24**, v3.4 P1) during a run (§2.10.2), so the handling ceiling was moved out with a matching physics posture rather than left in place: the mass-penalty coefficients (`CARGO_THRUST`/`CARGO_MAXSPD`/`CARGO_MASS`) were softened together (B-8) so a full chain at the (then) new ceiling reproduced the old full-12 feel, and `CHAIN_ITER` was bumped 3→4 for stability at 20 nodes (§3.4). v3.4 P1's cap raise 20→24 **deliberately did not** re-soften the coefficients — a bigger hold costs more handling by design (CS010 P2 firmed them further: **~37%/54% at m=24**, up from v3.4 P1's ~41%/58%) — but **did** re-validate the stability envelope at the new node count rather than assume it still held (§3.4). If the ceiling is ever raised again, follow that same pattern: decide explicitly whether to re-soften the coefficients, and always re-validate stability — don't just bump the number.
- **The tail-peel offload (chain.pop) matters visually** — the chain visibly shortens from the far end while the ship holds in the dock, which reads as "unloading." Popping from the head would make the whole chain teleport forward one link per tick.
- **Escalating per-piece scores (not a flat rate) is the whole risk/reward engine.** A 12-chain delivery (2,250) is worth far more than four 3-chain trips (450). This is what makes players tow long, dangerous chains past homing wedges. Flatten this and the mechanic collapses into an errand.
- **Aft-severing (losing everything behind a hit node) instead of single-node loss** makes chain *position* strategic: your most recent pickups are the most exposed. It also creates dramatic "lost the whole tail" moments that flat attrition wouldn't.
- **The dock relocates every wave** to prevent a dominant orbit-the-dock strategy and to keep the risk of the haul fresh each wave.
- **The dock is non-solid.** Making it collidable punishes the exact maneuver the game asks for (flying into it while handling badly). Hazards pass over it too — the dock is a zone, not cover.
- **The shield does not protect the chain.** The shield radius (26) only covers the ship; deflected debris can and will hit your own cargo. This is intentional — the shield answer and the cargo answer should be in tension, not stacked.
- **Momentum tug uses pre-constraint stretch** as its force signal (see 3.4). This is a stability decision: spring forces on a hard-constrained chain oscillate; measuring how much the cargo "resisted" the ship this frame and applying only that gives a heavy feel without energy injection.

### 2.11 Larger Scrolling World & Camera (v1.2, retuned v3.1 P1/P2)

The simulation space and the screen used to be the same 1280×720 rectangle. As of v1.2 they are decoupled: the world is a large torus and the screen is a camera window onto it.

- **World vs viewport.** The world is `WORLD_W × WORLD_H` = **2560×1440** (shrunk from the original v1.2 3840×2160 in v3.1 P1 — 2×2 viewports, 4× area ≈ 44% of the old world, cutting the dead space the bigger torus left between spawns) and still wraps at its own edges — same torus topology as before, just smaller. The **viewport** is `VIEW_W × VIEW_H` = **1280×720**, still the canvas size and still CSS-scaled/letterboxed to the window. All world/simulation math (wrap, distance, aiming, chain links) uses `WORLD_W/WORLD_H`; only the canvas, HUD, and title screen use `VIEW_W/VIEW_H`.
- **Camera.** `game.camera.x/y` tracks the ship's world position every frame — no smoothing, no clamping (the world wraps, so there is no edge for the camera to bump into). The ship is therefore always drawn at the center of the viewport. On respawn the ship returns to world center and the camera snaps with it.
- **Render pipeline.** `draw()` clears the viewport, draws the starfield, then (for gameplay) translates the context by `(VIEW_W/2 − camera.x, VIEW_H/2 − camera.y)` and draws every world-space entity inside that transform. The HUD (score, lives, cargo readout, dock chevron, wave, shield bar) and the pause/game-over overlays are drawn **after** restoring the untranslated context, so they stay screen-fixed. Each world entity is drawn at its **nearest wrapped image** relative to the camera (`wrapOffset`), so entities on the far side of the world seam still render in the right place when the ship is near a world edge (a single naive translate would drop them off-screen).
- **World-boundary line (v2.1) — removed (v3.6 P1b).** `drawWorldBoundary()` used to stroke a dim, dashed `WORLD_W×WORLD_H` rectangle (`COLOR.dim`, `setLineDash([BOUNDARY_DASH, BOUNDARY_GAP])`, low glow) as the first thing inside the camera transform — beneath every entity, drawn at the world-origin corner's nearest wrapped image relative to the camera so the seam stayed continuous as the ship crossed it. Purely cosmetic — an interim wrap-visibility aid, no collision or gameplay effect. **v3.6 P1b deletes it outright** — the function, its call site, and the four `BOUNDARY_*` constants are gone, with no replacement overlay. The point is to find out whether wrap reads fine without an explanatory guide; that's now a playtest question, not a shipped answer. (Full off-screen threat awareness remains deferred to v3.0 — see Culling.)
- **Culling.** `onScreen(e, margin)` (wrap-aware) skips drawing any entity more than `CULL_MARGIN` (100 px) outside the viewport. Culled entities still `update()` normally — off-screen hazards keep simulating in the larger world. (Off-screen *threat awareness* — warning the player about hazards approaching from beyond the viewport — is deliberately deferred; see `archive/PLANNED-FEATURES-v2.md` F10.)
- **Starfield (v3.0 P1: legibility + parallax).** Two layers. The **far layer** is unchanged in
  structure — stars scattered once across the whole world (count scales with world area,
  `STAR_DENSITY` raised 40→80), rendered wrap-aware in screen space at 1.0× the camera offset so it
  scrolls seamlessly with no tiling repeat or edge seam — but each star now carries its own
  brightness (`a` in `[STAR_BRIGHT_MIN, STAR_BRIGHT_MAX]`) drawn in a brighter cool blue
  (`rgba(120,160,210,a)`) than the old flat `#1a2a40`, so the field reads without competing with
  entities (Pillar 1). A second, sparser, brighter **near layer** (`starsNear`,
  `STAR_NEAR_DENSITY`, `rgba(160,195,235,a)`) adds a depth/motion cue: it moves at
  `STAR_PARALLAX_FACTOR` (0.5×) of the camera offset. Because a ≠1× layer is not world-fixed, it
  can't reuse the toroidal wrap math — instead it's tiled in **screen space**, modulo its own
  `VIEW_W×VIEW_H` tile, keyed off `camera * STAR_PARALLAX_FACTOR` (a 3×3 tile stamp each frame,
  culled to the viewport), which sidesteps seam artifacts entirely. Purely cosmetic — no gameplay
  effect either layer.
- **Ship-relative spawning.** Because a fixed-rect spawn could drop a whole wave unreachably far away in the big world, spawns are now placed relative to the ship's *current* world position: new large debris satellites in a ring `[SPAWN_MIN_DIST, SPAWN_MAX_DIST]` = **[220, 640]** px; the recycling dock in `[DOCK_MIN_DIST, DOCK_MAX_DIST]` = **[260, 620]** px (retuned in v3.1 P1 — see below). Killer satellites and saucers enter from just beyond a **viewport** edge relative to the ship (offset by `VIEW_W/2`/`VIEW_H/2` + a margin, then folded into the world with `wrapPos`), preserving their shipped "slide in from off-screen, cross the screen" feel; their speeds, homing, fire rates, and split behavior are unchanged (those are Phase 4/5 concerns).
- **Spawn-ring clamp (v3.1 P1).** On the smaller 2560×1440 world, a spawn ring reaching past half the shorter axis could wrap around and land on the far side of the ship — effectively on top of it. The binding limit is `min(WORLD_W, WORLD_H)/2 − 60 = 660`. `SPAWN_MAX_DIST` was clamped 1100→640 and `DOCK_MAX_DIST` 900→620 (mins unchanged at 220/260) so both rings stay safely under that limit, with `DOCK_MAX_DIST < SPAWN_MAX_DIST` preserved.
- **Starfield density preserved automatically.** `STAR_COUNT` derives from world area (`STAR_DENSITY * WORLD_W*WORLD_H / (VIEW_W*VIEW_H)`), so shrinking the world to ~44% of its old area scales the far-layer star count down to match — visual density is unchanged with no constant edits. The near (screen-space) layer is unaffected by world size.
- **Wrap jump is exactly `WORLD_W`/`WORLD_H` (v3.1 P2).** `wrap()` and the chain node-wrap (`wrapNode`) previously teleported entities by `WORLD_W + 120` / `WORLD_H + 120` — an undocumented +120 px mismatch against the period (`WORLD_W`/`WORLD_H`) every distance/aim/render helper (`dist2`, `angleTo`, `shortDelta`, `wrapOffset`, `wrapPos`) actually uses. That mismatch shifted every entity ~120 px relative to the ship at each seam crossing. Both jumps now teleport by exactly `WORLD_W`/`WORLD_H`; the ±60 px trigger margins are unchanged (they're anti-flicker hysteresis — a jump of exactly `WORLD_W` from `x = −61` lands at `WORLD_W − 61`, safely inside the `[−60, WORLD_W+60]` band with no re-trigger). `wrapNode` shifts `px`/`py` by the same delta as `x`/`y` so implied velocity survives the teleport (§3 rule).

> New tuning constants live in the "Larger world & scrolling camera (v1.2)" block at the top: `CULL_MARGIN`, `SPAWN_MIN_DIST`, `SPAWN_MAX_DIST`, `DOCK_MIN_DIST`, `DOCK_MAX_DIST`, `STAR_DENSITY`, plus `WORLD_W/H` and `VIEW_W/H`. v2.1 adds `BOUNDARY_DASH`/`BOUNDARY_GAP`/`BOUNDARY_WIDTH`/`BOUNDARY_GLOW` (the wrap-seam line) to the same block. v3.0 (P1) retunes `STAR_DENSITY` (40→80) and adds `STAR_PARALLAX_FACTOR`, `STAR_BRIGHT_MIN/MAX`, `STAR_NEAR_DENSITY`, `STAR_NEAR_BRIGHT_MIN/MAX` (the starfield legibility/parallax pass). v3.1 (P1) shrinks `WORLD_W/H` (3840×2160→2560×1440) and clamps `SPAWN_MAX_DIST` (1100→640) / `DOCK_MAX_DIST` (900→620) to match — `STAR_DENSITY` untouched, since `STAR_COUNT`'s area-derived formula auto-scales the far layer.

### 2.12 Health, Damage & Knockback (v1.3)

Discrete lives (3 stock + an extra life per 10,000 points) are gone. The ship carries a single HP pool for the entire run; survival is about managing that pool, not stock count. This keeps Pillar 4 intact — the shield is still the "free," energy-priced answer that prevents damage outright — while making one misjudged contact a costly setback rather than an instant stock loss.

- **HP pool.** `ship.hp` starts at `SHIP_MAX_HP` (250). **The HUD shows it as a ring, not a bar (CS009 P2, superseding the v3.6 P4 left-column HULL bar).** A top-right "ship status" element (§2.14 HUD layout) pairs the HULL ring with a concentric shield-energy arc (below) around a center ship glyph. The ring's value arc reads the **eased** `game.hudHull` (a hit/heal drains/fills over ~0.35s rather than snapping), but its **color** reads the **true, un-eased** `game.ship.hp` — the color tell flips the instant a hit lands even while the arc is still catching up. Three mutually exclusive states, precedence MAX > Critical > Normal: **MAX** (`hp >= SHIP_MAX_HP`) → `COLOR.ach` gold, solid, no pulse — `COLOR.ach` means exactly one thing everywhere in the HUD, "this resource is at its cap; more is wasted," and this ring is one of the two places it appears (the other is the CARGO ring's full/cap-up state, §2.10); **Critical** (`hp <= LOW_HP_THRESHOLD`, the same 40% threshold as the siren + low-HP chevron) → `COLOR.lowhp` red, and this is the one state that **pulses** (ring + center glyph together). **As of CS010 P3 the critical pulse is HP-ramped, not flat** — it reads `Math.sin(game.lowHpPhase)`, a shared phase accumulator advanced at `lowhpPulseRate(t)` (0.9→2.4 Hz with urgency `t`), so the hull ring, the corner glow (§ below), and the audio siren beat as **one instrument** at one rate. *(This reverses CS009's deliberate flat-`HUD_PULSE_HZ` choice — its reasoning was "the siren already ramps, so the visual stays flat," but two red HP tells at different rates beat visibly; Paul signed off on syncing them. `HUD_PULSE_HZ` now drives only the non-HP pulses — cargo-gold and the powerup low-timer.)* **Normal** (else) → `COLOR.hp` green.
- **Shield energy.** `game.ship.energy` renders as a second, thinner arc concentric with and inside the hull ring (`HUD_SHIELD_R_GAP` gap) — un-eased (energy is already continuous) and **never** state-colored; it's a separate resource from hull and must not get swept into a hull warning color.
- **Damage table** (contact = ramming a hazard body; every value is a named constant in the tuning block — first-pass and tunable):

  | Source | Damage | Constant |
  |---|---|---|
  | Small hazard (small debris satellite, small saucer) | 20 | `DMG_SMALL` |
  | Medium hazard (medium debris satellite, big saucer) | 35 | `DMG_MEDIUM` |
  | Large hazard (large debris satellite) | 50 | `DMG_LARGE` |
  | Hunter Satellite (large / medium / small) | 60 / 45 / 30 | `HUNTER_DAMAGE` |
  | Hostile (saucer) bullet | 15 | `DMG_BULLET` |

  Each hazard carries its own `damage` field, set in its constructor. Debris satellites map onto the shared tiers via the `DEBRIS_DAMAGE` size table; **Hunter Satellites (v1.6) carry their own higher per-tier values** (`HUNTER_DAMAGE` = 60/45/30) since they are the actively-seeking threat; saucers still use `DMG_MEDIUM`/`DMG_SMALL` by size. The shared `DMG_*` constants and the bullet value remain first-pass and tunable.
- **Knockback.** A non-lethal unshielded hit **sets** (not adds) the ship's velocity to `KNOCKBACK_SPEED` (250 px/s) pointing directly away from the hazard's center (wrap-aware, via `shortDelta`), so the ship physically separates instead of grinding inside the hazard. Setting rather than adding guarantees clear separation regardless of prior momentum; drag then bleeds the impulse off naturally over the following second.
- **Hit-stun.** The same hit sets `ship.invuln = HIT_STUN_DURATION` (1.0 s) of invulnerability (the existing blink render doubles as the hit-stun tell). Further hits are ignored until it expires, so no single hazard can chew through the pool in consecutive frames, and several hazards overlapping in one frame still only land one hit. `damageShip()` self-guards on shield/hit-stun/dead, so the collision passes don't need to break early.
- **Scoop decay (v3.3 P3).** A non-lethal unshielded hit is also what erodes the persistent **Scoop** upgrade (§2.14): after the HP/knockback/stat work, if `game.scoopLevel > 0` then `game.scoopHits++`, and every `SCOOP_HITS_PER_LEVEL` (2) hits drops one scoop level (`scoopLevel = max(0, −1)`), resets the tally, pushes a `"SCOOP −1"` float, and plays `AudioSys.scooploss()`. Because `damageShip()` early-returns on shield/hit-stun/dead, a **shielded or i-frame hit never counts** toward `scoopHits` — only real damage erodes the scoop. There is **no time decay**; the scoop goes away only by taking hits. The tally does **not** reset on a scoop pickup (FORK-3: no), and it does not accumulate at level 0 (hits there are harmless).
- **Death.** At 0 HP `killShip()` runs the explosion boom + particles, `scatterChain()` drops the whole tow load as free garbage, and the state goes straight to `gameover`. There is no respawn path — the old "respawn only when the center is clear" logic (former §2.1) is deleted entirely.
- **Shield unchanged.** A raised shield deflects/blocks at its usual energy cost, and a shielded hit takes **no** damage, knockback, or hit-stun — knockback is strictly the unshielded fallback. The shield still does not protect the tow chain (§2.10.1).
- **Low-health warning (v3.4 P5; corner glow + relief cue CS010 P3).** With no hit counter and variable damage per hazard, "running low" is expressed as an explicit HP threshold: `LOW_HP_THRESHOLD` (**100**, 40% of `SHIP_MAX_HP` — "five bad seconds from dead": 5 small hits / ~3 medium / 2 large). The **state** is a pure per-frame read, not a stored flag — `game.state === "playing" && !game.ship.dead && game.ship.hp <= LOW_HP_THRESHOLD` — so it engages and disengages freely; a single Health pickup (`POWERUP_HEALTH_AMOUNT` = 25) can clear it. **Four tells, all keyed on that one threshold and all edge-detected off the single `game.lowHpSiren` latch driven from one site in `update()`** (right after the camera-follow):
  - **Audio siren** — a pulsed, HP-scaled cockpit-warning voice, `AudioSys.lowhp(on)` + `AudioSys.lowhpSet(t)` (§2.8; rebuilt v3.5 P3 from the original constant-amplitude siren), engaged on the rising edge and disengaged on the falling edge or on leaving `playing` (`killShip`/`quitToTitle`/`openPause` each tear it down directly, mirroring `thrust`/`saucer`); `lowhpSet(1 − hp/LOW_HP_THRESHOLD)` is called every frame the state stays engaged, from the same site, scaling pulse rate/gain with urgency.
  - **Critical-hull ring pulse** — the top-right HULL ring turns `COLOR.lowhp` and pulses (§2.12 HP-pool bullet above).
  - **Corner glow (CS010 P3)** — four `createRadialGradient` corner fills in `COLOR.lowhp`, drawn first inside `drawHUD()` (so Capture's **H** hides it and **P** exports a clean frame), engaging on the exact same `game.lowHpSiren` predicate. A peripheral-vision alarm by design: low-frequency, large-area, **edgeless** — see §3.2 for why it is a *fill* and not a `glowStroke` arc (a named fills-exception). Intensity scales with **both** urgency `t` (brighter near death, `LOWHP_GLOW_ALPHA_MIN`→`_MAX`) **and** the shared pulse, so it gets brighter, not merely faster. `LOWHP_GLOW_*` are PLAYTEST KNOBS sized to **frame** the corners, not wash out the two occupied corners (the top-right HULL/CARGO rings, the bottom-left powerup rows).
  - **Directional chevron** — a second chevron (`COLOR.lowhp` = `#ff4040`, distinct from `COLOR.hp`, `COLOR.clumpHot`, and the low-HP bar fill `#ff7060`), same `drawPoly` shape as the dock chevron (§2.10) but orbiting screen-center at radius 58 (vs. the dock's 42, so both can show at once). It points at the **nearest** live `type === "health"` powerup (wrap-aware nearest-scan over `game.powerups`, via `angleTo`/`dist2`), drawn only while the state is engaged **and** a health powerup exists on the field.
  - **One instrument, one rate (CS010 P3).** The siren, the hull-ring pulse, and the corner glow all share **one** rate formula, `lowhpPulseRate(t)` = `LOWHP_PULSE_RATE_MIN + (MAX−MIN)·t` (0.9→2.4 Hz), read by `AudioSys.lowhpSet` (the LFO) **and** by a game-side phase accumulator `game.lowHpPhase`, advanced `(+= 2π·lowhpPulseRate(t)·dt) % TAU` in `update()` while engaged and read as `Math.sin(game.lowHpPhase)` by the ring and the glow. An **accumulator**, not `phase = ω·clock`, is mandatory: `ω` is HP-dependent, so `ω·clock` would jump the phase (~30 rad on a 0.5 rad/s Δ at t=60 s) each time HP changes and the pulse would stutter. The visual pulse **never** reads a Web Audio LFO value — those aren't reliably readable and `ctx` may be `null` before the first user gesture, so **the glow renders with audio unavailable**. `lowHpPhase` lives in `update()`, so it **freezes while paused** (correct — `openPause` tears the siren down anyway) and resets to 0 on disengage and in `startGame()`.
  - **Relief cue (CS010 P3, FLAG-CS010-a).** §5's relief cue was specced as Dan's voice line, but voice is a later, gated phase — so P3 ships a **visual** relief cue that stands on its own (if voice lands, Dan's line is *additive*, not a replacement). On the falling edge, **only if the ship is alive** (`game.hpReliefFlash = LOWHP_RELIEF_FLASH`), the HULL ring plays a **geometry-only** (radius + stroke-width) **non-pulsing** exhale that decays to normal over `LOWHP_RELIEF_FLASH` — the alarm *was* the pulse; the relief is its absence plus one clean exhale. It is deliberately color-untouched (on relief `hpCrit` is already false, so `hullColor` is back to `COLOR.hp`; the hpMax gold case still wins), so it composes with the state-color precedence chain and cannot leak `globalAlpha`. **The alive-guard is load-bearing:** dying also drives the low-health state false, so without `!ship.dead` the ring would play a cheerful "you're okay now" pop at the instant the player is killed. The flash decays in `update()` (beside `cargoFlash`, so `H`-hiding the HUD doesn't freeze it) and is zeroed at every direct siren teardown (pause / quit / start / death).
  - **FLAG-8a (open, deliberate — sharper as of v3.5 P3):** `POWERUP_HEALTH_GAP` (18–26 s ambient spawn cadence) was **not** shortened and no force-spawn was added for this warning. A player can sit in the low-health state with the alarm running and nothing on the field to point at for up to 26 s. The v3.5 P3 urgency scaling mitigates this somewhat, but a louder/more urgent alarm with nothing to point at near the end of that window is a sharper version of the same gap — still left open pending playtest, not force-fixed here.

> Tuning constants live in the "Health, damage & knockback (F2)" block at the top: `SHIP_MAX_HP`, `DMG_SMALL/MEDIUM/LARGE`, `DMG_BULLET`, `KNOCKBACK_SPEED`, `HIT_STUN_DURATION`, `LOW_HP_THRESHOLD` (100, v3.4 P5), `REPAIR_MILESTONE`, `REPAIR_AMOUNT`, `REPAIR_FULL_BONUS` (the per-size contact-damage map `DEBRIS_DAMAGE` lives with the debris constants). `COLOR.lowhp` sits with `COLOR` (§ palette). The low-health **voice** knobs (v3.5 P3) — `LOWHP_ROOT_FREQ`, `LOWHP_HARMONIC_RATIO`, `LOWHP_HARMONIC_GAIN_FRAC`, `LOWHP_PULSE_RATE_MIN/MAX`, `LOWHP_PULSE_DEPTH`, `LOWHP_GAIN_MIN/MAX`, `LOWHP_PARAM_RAMP` — sit in their own block right after `LOW_HP_THRESHOLD` (§2.8), followed (CS010 P3) by the shared `lowhpPulseRate(t)` helper and the **corner-glow / relief** knobs `LOWHP_GLOW_ALPHA_MIN/MAX`, `LOWHP_GLOW_RADIUS`, `LOWHP_GLOW_RGB` (mirrors `COLOR.lowhp`), and `LOWHP_RELIEF_FLASH` — all PLAYTEST KNOBS. `HUD_PULSE_HZ` (with the HUD `HUD_*` block) is now the **non-HP** pulse rate only (cargo-gold, powerup low-timer).

### 2.13 Difficulty Ramp & Early-Game Pacing (v1.5)

Implements Pillar 6 ("Ease players in", §1). One shared, wave-driven curve makes the opening waves approachable and lets intensity build on a schedule, tuned from a single knob.

- **The curve.** `difficultyFactor(wave)` = `1 − Math.exp(−(wave − 1) / RAMP_WAVES)` — **0 at wave 1**, climbing asymptotically toward (but never reaching) 1 over many waves. With `RAMP_WAVES = 8` (the one knob that sets how fast the whole game gets hard): wave 1 ≈ 0.00, wave 2 ≈ 0.12, wave 3 ≈ 0.22, wave 5 ≈ 0.39, wave 8 ≈ 0.58, wave 12 ≈ 0.75, wave 20 ≈ 0.91, plateauing near full by ~wave 25. Larger `RAMP_WAVES` = slower ramp (longer "enjoy the early game" window); smaller = steeper.
- **The interpolation helper.** `ramp(floor, ceil, wave)` = `floor + (ceil − floor) × difficultyFactor(wave)`. **Every** wave-driven threat parameter is expressed this way from a named floor (easy wave-1 value) / ceiling (full-intensity value) pair in the constants block, so the entire game's early-game pacing lives in one place. At wave 1 a parameter sits exactly on its floor; by the late waves it has closed most of the way to its ceiling.
- **What scales as of v1.5 — the saucers (§2.6):** spawn gap (`rand(20,30)s` floor → `rand(12,16)s` ceiling), fire-rate multiplier (`1.8×` → `1.0×`), small-saucer aim error (`±0.35` → `±0.09` rad), and small-saucer appearance chance (`15%` → `60%`, folding in the old >8000-point gate so escalation is wave-driven in one place, not two).
- **Small-saucer aim samples a *scaled* wave, not the raw one (CS012 P1).** The aim-error call is now `ramp(SAUCER_AIM_ERR_FLOOR, SAUCER_AIM_ERR_CEIL, 1 + (game.wave − 1) × SAUCER_ACCURACY_RAMP_SCALE)` — a new knob (`0.5` shipped) that lets saucer accuracy ramp independently of (and slower than) `RAMP_WAVES`, while pinning the wave-1 floor exactly (`SAUCER_ACCURACY_RAMP_SCALE × (wave − 1)` is `0` at wave 1, so the scaled wave collapses to `1` and the pre-CS012 wave-1 value is unchanged byte-for-byte). Every OTHER saucer parameter on this list (gap, fire-rate, appearance chance) still samples the raw `game.wave` — this is the one deliberately decoupled exception, per `PLANNED-FEATURES-CS012.md` FORK-CS012-A → (a).
- **What also scales (v1.6):** the **Hunter Satellites** (§2.5) wire every drift/homing speed and turn rate into this same `ramp`/`difficultyFactor` — their full-intensity numbers are the ceilings, `HUNTER_FLOOR_FRAC` sets the gentle wave-1 floor. So at wave 1 the whole Hunter family moves at ~60% speed, ramping up over ~20 waves.
- **What will scale next (not yet built):** the **Debris Satellite density** retune (the joint F3+F10 pass) still lands later; with garbage now permanent (v3.2 P3) it leans on coalescence + `GARBAGE_MAGNET_RANGE` and the drop-volume constants for density control (§2.10.1) rather than on decay or the difficulty curve.

> Tuning constants live in the "Difficulty ramp & early-game pacing (F10)" block at the top: `RAMP_WAVES`; the saucer floor/ceiling pairs `SAUCER_GAP_FLOOR_MIN/MAX`, `SAUCER_GAP_CEIL_MIN/MAX`, `SAUCER_FIRE_MULT_FLOOR/CEIL`, `SAUCER_AIM_ERR_FLOOR/CEIL`, `SAUCER_SMALL_CHANCE_FLOOR/CEIL`; and the saucer base fire ranges `SAUCER_FIRE_INIT/BIG/SMALL`. `SAUCER_ACCURACY_RAMP_SCALE` (CS012 P1, PLAYTEST KNOB) sits beside `SAUCER_AIM_ERR_FLOOR/CEIL` and scales only the aim-error wave argument, not the constant pair itself. The curve helpers `difficultyFactor` and `ramp` live in the Helpers block.

- **Difficulty levers (v3.4 P2).** A **lever** is a small, named, catalogued knob (`{enabled, start, floor}`) that scales one quantity from an easy wave-1 `start` down toward its shipped `floor` along this same `ramp`/`difficultyFactor` curve — levers never get their own curve. `leverScale(lever, wave)` (Helpers block, next to `ramp`) does the scaling; when `enabled` is false the lever is **inert** (pinned at `start`), so it ships built, wired, and tested without changing observed behavior. `floor` is a **hard clamp** — a lever can only ever bring a quantity back down to today's shipped baseline, never past it. The full registry (one entry per lever: what it scales, start/floor, enabled?, shipped version, playtest status) and the house rationale live in **`DIFFICULTY-LEVERS.md`** at the repo root — a living doc, read it before adding a new lever. v3.4 P2 seeded two, both shipped disabled: `LEVER_POWERUP_SIZE` (§2.14) and `LEVER_DOCK_SIZE` (§2.10).

### 2.14 Powerups (v1.7)

Six free-floating pickups (the `Powerup` entity class, standard contract) add temporary player buffs, an HP top-up, and — new in v3.3 P3 — a *persistent* collection upgrade. Each reads as a rotating hexagon housing (so they all register as one "collect me" family) wrapping a distinct vector-glow glyph, drawn via the shared `drawPowerupGlyph()` helper (reused by the HUD). Colors are hued apart for a readable tell (`POWERUP_COLOR`): Rapid amber, Triple magenta, Health red, Magnet blue, Engine mint, **Scoop violet** (v3.3 P3 — a hue with no other occupant in the palette, so it never reads as a hazard tell). **Two of the six are not timed effects:** Health is instant, and the **Scoop** (§2.14.1) is a *persistent, damage-decayed* ship upgrade — the first of its kind here — so it lives outside the `powerFx`/`powerBudget` machinery entirely.

- **Spawn sources.**
  - **Health** spawns *ambiently* on a saucer-like timer (`POWERUP_HEALTH_GAP` = 18–26 s between spawns), one at a time (its own `POWERUP_DECAY` bounds coexistence, like the one-saucer-at-a-time gate), in a reachable ring `[POWERUP_HEALTH_MIN_DIST, POWERUP_HEALTH_MAX_DIST]` = [220, 520] px around the ship. It reads as something that *shows up in the field*, not loot.
  - **Rapid Fire / Triple Shot / Magnet / Engine / Scoop drop from three deterministic sources (v3.6 P3, replacing the old per-small-kill chance roll):** every **Saucer kill** (large or small — `destroySaucer()`, the single choke point for the previously-copy-pasted player-bullet and shield-body-contact kill sites), every **large Hunter Satellite core** (`h.size === 3` only, *not* its medium/small children — the drop is the core "spitting out" its cargo), and the **recycle dock**, which emits one per visit the first time that visit's canister count passes through **10** (one line above the existing 12-canister Heavy Hauler latch — same "exactly once per visit" idiom). Each of the three calls the same unconditional `dropPowerup(x, y, vx, vy)` — the saucer/Hunter drops **inherit the killed entity's own velocity exactly** (no scaling, "flies on that vector"); the dock's drop launches from the dock's own position at `DOCK_POWERUP_SPEED` (120 px/s, a playtest knob) in a random direction, with a "SALVAGE BONUS" `FloatText` at the dock marking the cause. **Debris now drops nothing, ever** — this is a deliberate, flagged economy shift (§7 Balance/CONFLICT-3 in the phase history): supply moves from "any small-tier kill, 16% of the time" to "every UFO, every full Hunter lineage once, and every 10th delivered canister." **The type is still a *weighted* roll (unchanged since v3.3 P3)** over the **separate** table `POWERUP_DROP_WEIGHTS = {rapid: 3, triple: 3, scoop: 2, magnet: 1, engine: 1}` — favoring the shot powerups (weights are the playtest knob); `dropPowerup` just decides *what*, the three call sites decide *when*. Health is **not** droppable (it spawns ambiently). **⚠️ `POWERUP_DROP_TYPES` is NOT the drop table** — it is the list of *timed* effects the HUD active-effect row / `powerActive()` / `powerMode()` / `powerBudget` understand, and it deliberately **excludes Scoop** (which is persistent, not timed — FLAG A-9). The weighted drop table and the timed-effect list are two distinct structures, and this phase adds no new entry to either. Uncollected drops linger `POWERUP_DECAY` (26 s) before fading, and — as of v3.6 P3 — travel on their launch vector the whole time (the old "drop kick settles quickly" drag, `vx *= Math.pow(0.4, dt)`, is gone; a drop now flies until collected, expired, or wrapped, not just until it coasts to a stop).
- **Effects** — the four dropped types are *timed*; Health is *instant*. Rapid / Triple / Engine last `POWERUP_DURATION` (15 s); the **Magnet lasts `MAGNET_DURATION` = 30 s** (v3.4 P4 — the one long effect; a `powerDuration(type)` helper, mirroring `powerMode(type)`, returns 30 for magnet and 15 otherwise, so the shared `POWERUP_DURATION` is untouched for the other three):
  - **Rapid Fire:** raises the effective bullet cap from 4 to `RAPID_MAX_BULLETS` (8) while active (§2.2).
  - **Triple Shot:** fires 3 bullets in a narrow `TRIPLE_SPREAD` (0.14 rad) fan per shot and raises the cap to `TRIPLE_MAX_BULLETS` (12). **Rapid + Triple both active uses the higher cap (12), never the product (24)** — `maxBullets()` returns Triple's cap when Triple is up, regardless of Rapid.
  - **Health:** instant — restores +`POWERUP_HEALTH_AMOUNT` (25) HP, capped at max (§2.7). No timer.
  - **Magnet (buffed in v3.4 P4).** While active, free garbage within `MAGNET_RANGE` (**380 px** from ship center — deliberately past `VIEW_H/2` = 360, so a piece at the very top of the viewport still feels a pull) is drawn toward the ship by a real attraction force, and the effective pickup radius grows by `MAGNET_PICKUP_MULT` (1.6×). This 380 px reach replaces the old `MAGNET_RANGE_MULT × GARBAGE_PICKUP` ≈ 54 px — a ~7× range buff. **Strength has a quadratic falloff with a floor:** `t = 1 − d/MAGNET_RANGE` (1 at the ship, 0 at max range) and `accel = MAGNET_PULL_MIN + (MAGNET_PULL − MAGNET_PULL_MIN)·t²` — a dramatic near-field yank (`MAGNET_PULL` = **520** px/s² at zero distance) easing to a subtle nudge (`MAGNET_PULL_MIN` = **60** px/s², strictly > 0) at the edge, so even the top of the screen visibly drifts. `MAGNET_DAMP` (0.06) in-range velocity damping keeps the pull gliding, not jittering. **It now pulls *all* garbage, singles *and* clumps** (v3.4 P4 removed the old `pieces === 1` gate — v3.3's 9c made clumps directly scoopable, so pulling one is no longer noise). A clump is heavy, so acceleration is scaled by `accel /= √mass` (a mass-1 single is unaffected since √1 = 1; a mass-11 clump pulls at ~30% — it visibly drifts toward you but slowly enough to outrun or line up, preserving the "heavy clumps are slow anchors" identity). The pull uses the wrap-aware `shortDelta`/`dist2`, is a velocity nudge not a teleport (respects momentum, Pillar 2), and all distance/aiming math accounts for the toroidal world. **Duration is doubled** — see the expiry note below.
  - **Engine:** while active, halves the effective towed-mass sum returned by `chainMass()` (`ENGINE_MASS_MULT` = 0.5). Because that one function feeds the thrust penalty, top-speed penalty (both in `Ship.update`), the momentum-tug `massFactor` (in `updateChain`), and — via `shipTurnRate()` — the turn-rate penalty (CS010; dormant while `CARGO_TURN` = 0), they all lighten at once — "easier to maneuver even when towing garbage." Its whole magnitude rides on how much mass matters, so CS010 P2's mass retune (§2.10) is what makes Engine noticeable; `ENGINE_MASS_MULT` itself was left at 0.5 (Paul happy with the magnitude). Still a no-op with an empty chain (FLAG-4a, accepted).
- **Expiry modes (v3.0 P5 — configurable via §2.16 Options → Difficulty).** By default every timed effect expires on the `POWERUP_DURATION` clock (above). Two opt-in settings can swap the clock for a **use budget** on some effects: `shotPowerupMode` (`"time"` | `"shots"`) governs **Rapid Fire + Triple Shot** — a budget of `RAPID_SHOTS` / `TRIPLE_SHOTS` **trigger-pulls**, where a Triple 3-fan is **one** pull and, with both active, each budget decrements once per pull and ends **independently** (Triple can run out while Rapid keeps single shots going); `magnetMode` (`"time"` | `"pieces"`) governs the **Magnet** — a budget of `MAGNET_PIECES` **canisters hooked**, counted at the hook (not the pull), so a magnet that draws-then-hooks a piece spends only once. **Engine is always time-based** (not player-configurable); Health is always instant. Count budgets live in `game.powerBudget {rapid, triple, magnet}` (reset with the run, parallel to `powerFx`). A single predicate **`powerActive(type)`** resolves "is this effect live?" against the effect's mode (`powerFx` seconds in time mode, `powerBudget` count otherwise), and **`powerMode(type)`** returns the governing mode — the fire block, `maxBullets()`, `chainMass()`, the Magnet pull and the HUD all read through these so the two modes can never diverge. Both settings default to `"time"`, so shipped behaviour is unchanged until opted in.
- **Stacking rule — BANKS, does not refresh (v3.6 P4; reverses the earlier "refresh, never add" rule).** A same-type pickup while already active **adds** to what's left: `powerDuration(type)` (time mode: 30 s for magnet, 15 s otherwise) is added to the remaining `game.powerFx[type]` seconds, or `POWERUP_BUDGET[type]` is added to the remaining `game.powerBudget[type]` count — no ceiling, so a third or fourth pickup keeps banking. **Magnitude still never stacks** — two Rapid pickups do not double the fire rate; only duration/shot-count accumulates. Different types still run independently (Magnet + Rapid together is fine). Active-effect timers live in `game.powerFx` `{rapid, triple, magnet, engine}` (remaining seconds; 0 = inactive), counted down each frame in `update()`; count budgets live in `game.powerBudget`.
- **HUD (rebuilt CS009 P4/P5; rows made fixed/always-shown in CS012 P2 — see the HUD layout note below).** The powerup rows are rings, not bars, stacked in a bottom-left column: **as of CS012 P2, one row per TYPE** in the stable `POWERUP_DROP_TYPES` order (rapid/triple/magnet/engine — this order is load-bearing, never inlined), **always drawn regardless of `powerActive(t)`** — activating or expiring a type no longer moves any row, because nothing is compacted. Each row is a dim full-circle track (always drawn, so every row reads as the same shape) plus the type glyph dead center plus a label/number pair to the right. An **inactive** row is muted: glyph/label/number in `COLOR.dim`, no value arc, no low-timer pulse, no bank pop; its number reads the idle `"0s"`/`"0"` the existing expression already produces (no special-casing). An **active, time-mode** row gets a value arc, `remaining / powerDuration(type)` (the magnet denominator is 30s, not the shared 15s — `powerDuration()` is called, never the constant directly, or the magnet ring would render permanently full), clamped to one full turn — a banked surplus (`frac > 1`, from the stacking rule below) draws as a second, thinner **overcharge halo** ring just outside the main one (`clamp01(frac - 1)`) instead of being silently clamped away. **Active, count-mode rows draw no value arc at all** — just the dim track, glyph, and the raw remaining budget number, since a budget has no fixed denominator to draw against. An active time-mode row within `HUD_FX_LOW` (3s) of expiring shifts ring + glyph + label + number to `COLOR.lowhp` and pulses (`HUD_PULSE_HZ`); count mode never state-colors this way (a budget isn't a deadline). **Banking pops the ring (CS009 P5), active rows only.** A same-type pickup while already active — a bank, per the stacking rule below — flashes a second, larger ring (a 25% pop settling over `HUD_BANK_FLASH` 0.6s) plus a fading `+Ns` (bare `+N` in count mode) badge beside the number, so banking has a visible tell instead of vanishing into what used to be a bar. Health, being instant, never appears in this stack. A `FloatText` label and a rising two-note `AudioSys.powerup()` chime still confirm each pickup.
- **HUD layout (rebuilt CS009 P0–P6; CS012 P2 fixed the bottom-left stack's row count/positions — see below).** Three zones, not one column: **top-left** is just Score → Level (§2.7); **top-right** is the HULL ring + concentric shield arc (§2.12) beside the CARGO ring (§2.10); **bottom-left** is a **fixed FIVE-row stack** anchored at `HUD_FX_BASE_Y`, all five rows ALWAYS drawn (CS012 P2 — no compaction, ever): row 0 (`y = HUD_FX_BASE_Y`) is the persistent **Scoop segmented ring** (§2.14.1); rows for the four timed types sit at `HUD_FX_BASE_Y - (i+1)*HUD_FX_ROW_H` for `i` = the type's index in `POWERUP_DROP_TYPES`, active or not — a type's row never moves regardless of what else is (in)active. Two chevrons orbit screen-center independently of this stack: the dock pointer (r=42, hauling-only, §2.10) and the low-HP pointer (r=58, §2.12). Every element strokes through `drawRingArc`/`drawRingSegments`/`glowStroke` like the rest of the game (§3.2) — the only surviving HUD fill in this stack is `drawText`. Element positions are named `HUD_*` constants (`HUD_HULL_CX`/`HUD_CARGO_CX`/`HUD_RING_CY`/`HUD_FX_BASE_Y`/`HUD_FX_ROW_H`/`HUD_RING_SEG_GAP`/etc.), never bare literals.
- **Pickup / lifecycle.** Collected when the ship comes within `p.radius + SHIP_RADIUS` (**not** the raw `POWERUP_RADIUS` constant — see the size-lever note below). Uncollected powerups fly on their launch vector (no drag, v3.6 P3 — ambient Health, which spawns at rest, is unaffected either way), screen-wrap, and fade after `POWERUP_DECAY` (26 s) with the same blink countdown as garbage. Drawn above the hazards in z-order so they stay readable.
- **Low-health pointer (v3.4 P5).** While the ship is at or below `LOW_HP_THRESHOLD` (§2.12) and a Health powerup exists on the field, a red chevron points at the nearest one — see §2.12 for the full mechanic (it's specified there since it's a symptom of the health/damage system, not a powerup-system feature).
- **Size is a difficulty lever (v3.4 P2, ships DISABLED).** `Powerup.radius` is `POWERUP_RADIUS * leverScale(LEVER_POWERUP_SIZE, game.wave)`, baked once per pickup at construction. The **pickup collision gate reads `p.radius`**, not the `POWERUP_RADIUS` constant, so the collision circle can never silently disagree with the rendered hex housing (the draw path already scaled off `this.radius`, unchanged). With the lever disabled, `LEVER_POWERUP_SIZE.start` (2.0) is pinned at every wave, so every powerup is simply **2× its old size**. See §2.13 and `DIFFICULTY-LEVERS.md`.

> Tuning constants live in the "Powerups (F6)" block at the top: `POWERUP_DURATION` (15), `MAGNET_DURATION` (30 = 2× `POWERUP_DURATION`, v3.4 P4), `POWERUP_RADIUS`, `POWERUP_DECAY` (26), `POWERUP_DROP_TYPES` (the *timed-effect* list, not the drop table), `POWERUP_DROP_WEIGHTS` (the v3.3 P3 weighted drop table — now rolled by all three v3.6 P3 emitters), `DOCK_POWERUP_SPEED` (120, v3.6 P3 — the recycle hub's per-visit launch speed), `POWERUP_HEALTH_GAP`, `POWERUP_HEALTH_AMOUNT`, `POWERUP_HEALTH_MIN/MAX_DIST`, `RAPID_MAX_BULLETS`, `TRIPLE_MAX_BULLETS`, `TRIPLE_SPREAD`, `MAGNET_RANGE` (380, v3.4 P4 — replaced `MAGNET_RANGE_MULT`), `MAGNET_PICKUP_MULT`, `MAGNET_PULL` (520), `MAGNET_PULL_MIN` (60, v3.4 P4), `MAGNET_DAMP`, `ENGINE_MASS_MULT`, the v3.0 P5 count-budget knobs `RAPID_SHOTS` (40), `TRIPLE_SHOTS` (30), `MAGNET_PIECES` (**40** as of v3.4 P4, was 20) + the `POWERUP_BUDGET` lookup, and the v3.3 P3 **Scoop** block `SCOOP_MAX_LEVEL` (5), `SHIP_DRAW_W` (18, v3.4 P3), `SCOOP_CONFIG` + `buildScoopSteps` (v3.4 P3 — generate `SCOOP_WIDTH`/`SCOOP_DEPTH`, the per-level mouth geometry), `SCOOP_HITS_PER_LEVEL` (**5** as of v3.4 P3, was 2), `SCOOP_MAX_BONUS` (500). **`POWERUP_DROP_CHANCE` is gone (v3.6 P3)** — there is no chance gate left; the three sources above are unconditional. `POWERUP_COLOR`/`POWERUP_LABEL` sit with `COLOR`; the `Powerup` class + `drawPowerupGlyph` are in the Entities block; `powerMode`/`powerActive`/`powerDuration`/`inScoopBox`/`maxBullets`/`dropPowerup`/`destroySaucer`/`spawnHealthPowerup`/`applyPowerup` are Flow functions. The two expiry-mode settings (`shotPowerupMode`, `magnetMode`) live on the persisted `settings` object (§2.16).

#### 2.14.1 The Scoop — a persistent, damage-decayed collection upgrade (v3.3 P3)

Every powerup before the Scoop was either instant (Health) or timed (Rapid/Triple/Magnet/Engine). The **Scoop** is the first **persistent** upgrade: it lives on **`game.scoopLevel` (0…`SCOOP_MAX_LEVEL` = 5)** and **`game.scoopHits`** (a damage tally), both reset in `startGame()`, and it is deliberately **not** in `game.powerFx` or `game.powerBudget` — there is no clock on it.

- **Gaining levels.** Picking up a `"scoop"` drop calls `applyPowerup("scoop")`, which raises `scoopLevel` by 1 (capped at 5) and pushes a `"SCOOP +1"` float. At the cap it instead pays a small `SCOOP_MAX_BONUS` (500) score bonus and floats `"SCOOP MAX"` — mirroring the full-HP score-milestone pattern (§2.7) so a pickup at max is never wasted. A scoop pick still counts as a powerup (`game.stats.powerupsPicked++`), so — like any powerup — **it freezes the `maxWaveNoPowerup` "No Powerups Needed" counter** (FLAG A-8: consistent, it *is* a powerup). It does **not** feed `powerUsed[]` (that gates "Powered Up", the four timed types only).
- **The capture mouth (geometry) — a GENERATED config (v3.4 P3).** Each level widens/deepens an **oriented capture box** on the ship's facing axis (FORK-2 resolved: box, not a circle — a directional *mouth* you sweep garbage into, not a bigger radius). `SCOOP_WIDTH`/`SCOOP_DEPTH` were originally hand-picked literals; as of v3.4 P3 they're **generated** by `buildScoopSteps(min, max, curve)` from a `SCOOP_CONFIG` object — `maxWidthMult`/`minWidthMult` (L5/L1 mouth width, as a multiple of the new `SHIP_DRAW_W` = 18, the drawn hull's width), `curve` (1.0 = linear step distribution, >1 weights growth toward the top levels), and `minDepth`/`maxDepth` (px forward of ship center at L1/L5). `buildScoopSteps` computes step `k` as `min + (max-min) * ((k-1)/(N-1))**curve` for `k` in `1..SCOOP_MAX_LEVEL`, with index 0 hardcoded to 0. **The load-bearing invariant `SCOOP_WIDTH[0] === 0 && SCOOP_DEPTH[0] === 0`** is asserted at load (not just tested) — it's what makes `inScoopBox` return false at `scoopLevel` 0, so pickup stays byte-identical to the pre-scoop build. Shipped config as of v3.6 P1c: `{maxWidthMult: 5.0, minWidthMult: 1.2, curve: 1.0, minDepth: 20, maxDepth: 60}` — `maxWidthMult` **3.0→5.0** (L5 mouth width 54px→**90px**, 5× `SHIP_DRAW_W`) and `maxDepth` **36→60** (a proportional bump; depth wasn't specified alongside width, so this is a playtest starting point, not a settled decision — re-pick both in `tools/scoop-lab.html`). The generated arrays keep the exact same shape (length `SCOOP_MAX_LEVEL+1`, index = level) as the old literals, so `inScoopBox`, `Ship.draw()`'s prong V, and the HUD pip row all read them unchanged. **`tools/scoop-lab.html`** is the disposable design instrument for picking these five numbers: a standalone (no build step, no imports — it carries its own duplicate copies of `buildScoopSteps`/`inScoopBox`/`drawPoly`/`glowStroke`/the ship hull poly, since drift there can only ever produce a bad *preview*, never a bad *build*) interactive harness with live sliders, a level stepper + "show all 6 levels" ghost overlay (so the scaling *between* steps is visible), a draggable scatter of real-size (radius-7) garbage canisters highlighted when inside the real `inScoopBox` capture region, a readout of the generated arrays, and a "copy config" button that prints the exact `SCOOP_CONFIG` literal to paste into the game. It answers "how big does this look," not "how does this play" (no thrust/chain/wrap/clumps).
- **Losing levels — decay by damage, not time.** The scoop erodes only through the `damageShip()` non-lethal-hit path (§2.12): every `SCOOP_HITS_PER_LEVEL` (**5** as of v3.4 P3, up from 2 — a playtest knob) hits drops one level. A shielded/i-frame hit doesn't count (damageShip early-returns); there is no time decay. This makes the scoop a *standing* reward for flying clean — it grows as you finish lineages and shrinks as you take hits. **At 5 hits/level a level-5 scoop survives 25 non-lethal hits**; with `SHIP_MAX_HP` 250 and `DMG_MEDIUM` 35 a full-health run only takes ~7 hits total before death, so the scoop is now effectively *sticky* once earned — that's the intent of the P3 retune, not a bug for a future session to "fix."
- **Render — the prong-V, restored (CS010 P1; supersedes the v3.6 P1c capture box — history below).** `Ship.draw()` draws an open `drawPoly` V, `[[d,-hw],[16,0],[d,hw]]` (throat at the nose, flaring forward to the mouth's two outer corners), in `COLOR.dock`, **no fill**. This is a render-only change: `inScoopBox()`'s capture math — the full rectangle from `forward = -SHIP_RADIUS` (behind the nose) to `forward = SCOOP_DEPTH[lvl]`, `|lateral| ≤ SCOOP_WIDTH[lvl]/2` — is untouched and was never conditioned on what's drawn.
  - **History.** Through v3.5 the scoop was drawn as this same V. v3.6 P1c replaced it with a translucent-fill rectangle — corners `[[-SHIP_RADIUS,-hw],[d,-hw],[d,hw],[-SHIP_RADIUS,hw]]` — drawn as the *exact* box `inScoopBox()` tests, specifically because the V never showed the box's rear edge (below). That fill made it §3.2's no-fills exception (1). CS010 P1 reverts to the V and **drops that fill exception** (§3.2 exception count 3→2, see §3.2): Paul decided the cleaner open-prong look is worth the incomplete tell. **This bullet has been rewritten three times across two supersessions (V→box→V). If a future session is tempted to "fix" it back to the box because it doesn't show the rear edge, that trade was already made knowingly — don't re-decide it, ask Paul first.**
  - **What's given up.** The V's arms run from the nose `(16,0)` forward to the mouth's outer corners at `(d, ±hw)` — it never depicts the box's rear edge, the segment at `forward = -SHIP_RADIUS` (behind the nose), which is very much part of the live capture region. A canister drifting into that rear strip is captured with no visual tell that it was ever "in" anything. That was the stated reason v3.6 P1c replaced the V; CS010 P1 knowingly reintroduces it.
  - `tools/scoop-lab.html` carries its own duplicate V render (per the tool's disposable-instrument convention) so the lab previews what the game actually draws.

Still a persistent-state readout, **not** a timed effect — deliberately drawn *outside* the `POWERUP_DROP_TYPES`/`powerActive` loop.

### 2.15 Controller Input (v1.8)

Gamepad support layered onto the existing keyboard input with **no change to any call site** — everything downstream still calls `input.left()`, `input.thrust()`, etc. The input layer now resolves each action against *either* the keyboard *or* the first connected gamepad, driven by a single data-driven binding table so Phase 8's rebinding UI has one structure to edit. **These are hardcoded defaults; the rebinding UI itself is F8/Phase 8.**

- **Binding table (`bindings`).** One entry per action, expressed as data — not per-action logic: each lists its keyboard `keys` (lowercased, matching the `keys{}` map), gamepad `buttons` (Standard-Gamepad indices from the `GP` enum), an optional analog `axis` (`{index, dir}`, where `dir < 0`/`> 0` is the negative/positive half past the deadzone), and a `fixed` flag (F7 spec: gameplay actions are rebindable → `fixed:false`; menu/system actions are non-configurable → `fixed:true`).
- **Gameplay defaults** (rebindable in F8): **D-Pad Left / Left Stick X < −0.25 → rotate CCW**; **D-Pad Right / Left Stick X > 0.25 → rotate CW**; **D-Pad Up / Left Stick Y < −0.25 → thrust**; **A → fire**; **Right Trigger *or* Right Bumper → shield**. The stick threshold is `GP_DEADZONE` (0.25, the one controller-*feel* tuning knob), applied with a strict `>` so exactly-`0.25` does not register. *The shield binding was not specified in F7 — right trigger/bumper is a deliberate gap-fill (see the delivery notes / STATUS.md); everything else is Paul's spec.*
- **Menu / system reads** (fixed, wired now for later phases): **Start → start / pause / unpause** (mirrors Enter on title/gameover and `P` while playing), **A → confirm** (title/gameover only, so it never collides with A = fire during play), **B → back** (reserved — no menu screen to back out of until F8). These are **edge-triggered** (fire once per press, not every held frame) via `gpPressed()`, which compares the current button snapshot against the previous frame's.
- **Polling.** `pollGamepad()` runs once per frame in the main loop, *before* `update()`. It reads `navigator.getGamepads()`, takes the **first connected** pad (skipping empty `null` slots) and ignores any others, and snapshots its buttons/axes (plus the prior frame's buttons for edge detection). `handleGamepadMenu()` runs right after it to process the edge-triggered menu reads. Held-state gameplay actions are resolved lazily inside `input.*()` from that snapshot, so `Ship.update` reads them exactly as it reads the keyboard.
- **Audio unlock caveat.** A controller Start/A press makes a best-effort `AudioSys.init()`/`resume()` (the same call the keyboard does on keydown). Browsers do not reliably treat gamepad input as a user-activation gesture for the Web Audio autoplay policy, so a controller-only player may still need one keyboard press to enable sound — flagged as a playtest item, not a code bug.

> Tuning constant: `GP_DEADZONE` (0.25) in the "Controller / gamepad input (F7)" block at the top. The `GP` button/axis-index enum, the `bindings` table, `pollGamepad`/`handleGamepadMenu`, and the resolution helpers (`kbActive`/`gpActive`/`actionHeld`/`gpPressed`) all live in the Input section; `input` and the keyboard listeners are unchanged in shape.

### 2.16 Pause Menu, Options & Control Rebinding (v1.9)

The game's first real menu/UI state machine, layered on top of the F7 input layer (§2.15) so every screen is navigable by keyboard **or** the first connected gamepad. It turns v1.8's hardcoded `bindings` into player-editable ones and gives the volume sliders something to drive.

- **State machine.** `game.paused` is true **iff** a pause-menu screen is open; `game.menu.screen` names it: `null` (closed) → `"root"` → `"options"` (SFX / Music / Master sliders + **Music Track** + Controls + Achievements + Difficulty + Back) → `"controls"` (per-action rebind grid + Return to Defaults + Back), plus `"achievements"` (read-only viewer, §2.17) and `"difficulty"` (powerup-expiry mode toggles, v3.0 P5 — below). `game.menu` also carries `index` (cursor for root/options), `row`/`col` (cursor for the controls grid: one row per rebindable action, col 0 = keyboard, col 1 = gamepad; plus Return-to-Defaults and Back rows), `rebinding` (`{action, device}` while capturing, else `null`), and `achReturn` (`"root"` | `"options"` — which parent the Achievements viewer backs out to, since it has two entry points). **Quit returns to the title screen** (a webpage can't reliably close its own tab).
- **Opened from any state (v3.0 P4 — supersedes B-1-a; closes the long-standing "menu reachable only via pause").** `openPause()` is now callable from `title`/`gameover` as well as `playing`; `update()` already skips the sim outside `playing`, so `game.paused` is safe in any state. The **root layout is context-aware**, keyed purely on `game.state` (pausing mid-game keeps state `"playing"`; the system menu keeps `title`/`gameover`): `rootItems()` returns `MENU_ROOT_PLAY = ["Continue","Options","Quit"]` when playing, else `MENU_ROOT_SYS = ["Options","Achievements","Back"]`. `menuRoot()` dispatches **by label** so one handler serves both layouts; "Back" (system root) and "Continue" (play root) both `closePause()`. The Options/Controls/Achievements sub-tree is reused **unchanged**; the only sub-tree tweak is `achReturn`, so the Achievements viewer backs to the system root when entered there and to Options when entered via Options. The panel title reads "PAUSED" mid-game, "MENU" for the system menu.
- **Input routing & separation (the core correctness concern).** Both input methods funnel discrete actions (`up/down/left/right/confirm/back/pause`) into one `menuInput(action)` dispatcher — keyboard via the `keydown` handler (arrows/WASD nav, Enter confirm, **Esc back**, read from the same `bindings.confirm/back/pause` entries), gamepad via `handleGamepadMenu()` (edge-detected D-Pad/Left-Stick nav through a `menuNav` latch; A confirm / B back / Start resume). Both handlers check **three contexts in priority order**: (1) **rebind capture** — the pressed key/button *is* the new binding and is consumed entirely (never written to `keys{}`, never routed to nav); (2) **menu open** — input drives navigation only; (3) **normal play/title/gameover**. Because gameplay reads (`input.left()` …) happen only inside `update()`, which early-returns while `paused` (or outside `playing`), **menu input can never move the ship**.
  - **v3.0 P4 — the two newly context-dependent inputs.** `bindings.pause.keys = ["escape"]` (P retired, FLAG P4-a) and `bindings.back.keys = ["escape"]` **share** ESC: in-menu, `handleMenuKey` resolves **confirm → back → pause** in order, so ESC (matching `back` first) means "back"; outside a menu, the normal branch's `pause`-during-`playing` check makes ESC pause. Controller **Start** is the session toggle (`playing → openPause`, else `startGame`; while paused the menu branch routes Start → `menuInput("pause")` = dismiss/resume). Controller **B** mirrors ESC's dual meaning: menu-open → `back`; no menu on `title`/`gameover` → `openPause()` (the system menu). Keyboard **"O"** is B's keyboard twin for opening the system menu (handled inline in the normal branch, not the binding table). In the gamepad normal branch, Start/B/A are an **else-if chain** so one frame can't both open a menu and start a game (**FLAG P4-b**); on keyboard the menu-open context returns before the title "press to start" line, giving the same guarantee.
  - **Keyboard menu repeat guard (v3.5 P1).** The menu-open branch of the `keydown` listener now ignores `e.repeat` (still calling `preventDefault()` so the key doesn't scroll the page), matching the gamepad path — `handleGamepadMenu()` was already strictly edge-detected via `menuNavEdges()`. Before this, browser key auto-repeat (~30/s while held) fired `handleMenuKey` on every repeat tick; on a small cycling row (e.g. Music Track) a brief hold could spin the selection back to where it started ("nothing changed" from the player's view) while re-entering `MusicSys.setState()` nearly every frame, discarding and rebuilding `trackGain` and restarting its 0.6 s crossfade from `0.0001` — which is what made the music sound like it had stopped. The guard also applies to the three volume sliders and the two Difficulty toggles, since they're driven by the same menu-open branch: **a held key no longer scrubs them** (a full 1.0→0.0 volume sweep is now 10 taps, matching gamepad behavior) — a deliberate, logged consequence, not a bug; a throttled-repeat softening is a possible playtest follow-up, not added here. The rebind-capture branch and the normal-play branch are untouched — gameplay keys are read from `keys{}` continuously and legitimately want held-key behavior.
- **Volume.** `AudioSys` routes every voice through an SFX gain → master gain bus (+ the music gain, fed by `MusicSys`, and — **CS010 P9** — a voice gain, fed by `VoiceSys` — §2.8), so the sliders attenuate real output. Values live on `AudioSys.vol {master,sfx,music,voice}` (0..1, clamped), applied to the live gain nodes by `setVol()`; the sliders nudge by `VOL_STEP` (0.1). **The Voice Volume row (CS010 P9)** sits on the Sound/Music sub-screen after Master Volume, added by growing the paired `SOUND_ROWS`/`VOL_LABELS`/`VOL_CATS` arrays together (`voice`); it persists additively into the frozen `afd_settings_v1` like every slider before it.
- **Music Track (v3.4 P6, track set replaced v3.5).** A left/right cycle row placed directly after "Master Volume", over the four values `["Zen","Derelict","Drift","Warehouse"]` (calm → hot; the same toggle idiom as the Difficulty screen), persisted as `settings.musicTrack` (default **"zen"**). It selects which gameplay track `MusicSys` plays during `playing` (§2.8); changing the row mid-game (from the pause menu) crossfades live, since `musicStateFor()` reads `settings.musicTrack` every frame. The title track (Beacon) always plays on the title screen regardless of this setting.
- **Voice & Captions (CS011 P3).** Two rows on Sound/Music, between "Voice Volume" and "Music Track": **Voice**, a left/right cycler over the seven `VOICE_STYLE_VALUES` (`"off"` + the six `VOICE_STYLES` — Comms/Comms ♀/Monotone/Monotone ♀/Computer/Computer ♀, §2.8), wrapping both directions and calling `VoiceSys.setStyle()` on every change so the switch is audible immediately; and **Captions**, a left/right toggle (either direction flips it, same as the volume-slider idiom) showing "On"/"Off", governing `game.caption`'s on-screen render (§2.8) independently of Voice Volume — captions still show with Voice set to Off. Both persist as `settings.voiceStyle`/`settings.captions`. `drawSound()`'s panel grew (510→602px tall) to fit the two new rows; both are plain `drawText` value columns, the same layout `Music Track` already used — no new fill/stroke idiom.
- **Label-dispatch refactor (v3.4 P6).** `menuOptions()` and `drawOptionsMenu()` previously dispatched on **hardcoded row indices** (`if (m.index <= 2)` for the sliders; `=== 3/4/5/6` for Controls/Achievements/Difficulty/Back), so inserting a row would silently shift every branch. Both now dispatch **by label** (the precedent `menuRoot()` already set), reading `MENU_OPTIONS[m.index]` and matching on the string — so the new Music Track row was purely additive and the index-fragile code no longer exists. The slider rows resolve their category via `VOL_LABELS.indexOf(label)` → `VOL_CATS`, not a bare index.
- **Rebinding.** Selecting a binding cell and confirming arms capture for that `(action, device)`; the next key (keyboard) or button (gamepad) rewrites `bindings[action].keys`/`.buttons`. A **gamepad** rebind replaces the button set and **drops the analog axis** for that action (so the displayed binding matches what actually fires — a stick direction can't be captured, only a button). Esc cancels a capture. **Return to Defaults** restores the five rebindable actions to `DEFAULT_BINDINGS` — a pristine snapshot taken at load, *before* saved settings are applied. Only the F7 gameplay actions (`fixed:false`: rotate CCW/CW, thrust, fire, shield) are editable; menu/system actions (A/B/Start) stay fixed.
- **Difficulty screen (v3.0 P5, B-4/B-5).** A `"difficulty"` sub-screen reached from Options → Difficulty (mirrors the tested `menuOptions`/`drawOptionsMenu` pattern via `menuDifficulty`/`drawDifficulty`). Two left/right toggle rows — **"Shot powerups expire: Time | Shots"** (`settings.shotPowerupMode`) and **"Magnet expires: Time | Pieces"** (`settings.magnetMode`) — plus Back; ◄ selects the timed default, ► the count variant. Flipping a toggle persists immediately (`saveSettings`). These two settings drive the powerup expiry logic in §2.14; both default to `"time"`. The screen is deliberately structured to grow (a `DIFFICULTY_ROWS` list + a switch case + a renderer row) as future difficulty knobs land (FLAG B-4-c).
- **Persistence.** Volumes + custom bindings **+ the two difficulty modes** (`shotPowerupMode`, `magnetMode`) **+ the selected music track** (`musicTrack`, v3.4 P6) **+ the voice style and captions toggle** (`voiceStyle`, `captions`, CS011 P3) save to `localStorage` (`afd_settings_v1` — the `afd_` prefix is **frozen**, v3.3 P1: renaming it to match "Orbital Overhaul" would silently wipe every player's saved settings/bindings, so it stays `afd_*` regardless of the product rename) on every change and load once at startup, over the shipped defaults. The v3.0 P5 modes, the v3.4 P6 `musicTrack`, and the CS011 P3 `voiceStyle`/`captions` are all **additive fields** — no schema-version bump: an older save missing them simply keeps the runtime default (`"time"` / `"zen"` / `"comms"` / `true`), and a corrupt/unknown value is ignored on load — `musicTrack` is accepted only if it is one of the **four current** `MUSIC_TRACK_VALUES` (`zen`/`derelict`/`drift`/`warehouse`); `voiceStyle` only if it's a real `VOICE_STYLE_VALUES` id; `captions` only if it's literally a boolean (a stray string like `"yes"` is ignored, not coerced — the true default can't accidentally be knocked to a truthy-but-wrong value). **A save from a build predating v3.5 gets a one-time fallback, not an upgrade path:** `"retro"`/`"tense"`/`"ambient"` (all retired names from the v3.4 P6/P7 track set) fail that check and `loadSettings` silently falls back to the shipped default `"zen"` — no crash, no key migration, and no attempt to map an old name onto its nearest new-set equivalent; the player just lands back on the default track and can re-pick from Options. **All storage access is `typeof`-guarded and try/caught** — `localStorage` is unavailable in the claude.ai artifact sandbox (and can throw in privacy modes), and a storage failure must never crash the game (GDD §4 item 1). *Persistence itself still needs verifying in a real browser outside the sandbox — see STATUS.md.* **This is one of three independent `localStorage` keys, not the whole persistence story:** `afd_settings_v1` (here), `afd_achievements_v2` (§2.17), and `afd_scores_v1` (§2.18) are separate stores with separate guarded `storageOK()` load/save paths — none reads or writes either of the others, and all three names are frozen (see `CLAUDE.md`).
- **Menu restructure (CS010 P4, §8b/§9/§10a) — a relocation, not a redesign.** `MENU_OPTIONS` is now `["Sound / Music","Controls","Achievements","High Scores","Difficulty","Back"]`. The three volume sliders + Music Track moved **off** Options onto a new `"sound"` sub-screen (`SOUND_ROWS` — as of CS010 P9 + CS011 P3, `["SFX Volume","Music Volume","Master Volume","Voice Volume","Voice","Captions","Music Track","Back"]` — driven by `menuSound`/`drawSound` — the same label-dispatch template as `menuDifficulty`/`drawDifficulty`; `VOL_LABELS`↔`VOL_CATS` stay a paired lookup, independent of either screen's row order), leaving `drawOptionsMenu` a plain centered nav list like `drawRootMenu`. **"High Scores" moved the other way — off `MENU_ROOT_SYS` and onto Options (FORK-4).** Nesting it under Options makes the table reachable from **both** the system menu (title/gameover) **and** the pause menu mid-game, which the old root-only row never was — the entire point of §8b; `menuHighScores` now backs to `"options"` via `MENU_OPTIONS.indexOf("High Scores")`, and with a single entry point no `scoreReturn` tracker (mirroring `achReturn`) is needed. **FLAG-8b fixed as part of the same pass:** the two remaining hardcoded `gotoScreen("options", 3/4)` sites (`menuControls`'s two branches, `menuAchievements`'s one) now resolve via `MENU_OPTIONS.indexOf(...)`, so no index-fragile call site survives anywhere in the menu tree. P2's *Ship Rotation* row on Controls (below) is untouched by this restructure.

### 2.17 Achievements (v2.0; tiered lifetime v3.0 P7; new tiered/event/weekly v3.0 P8)

A progress-tracking layer that **observes** stats the rest of the game already produces and never drives gameplay — the widest-reaching but lightest-touch system, all funnelled through one `Achievements` module (defined after the powerup flow functions). Two pools, 35 achievements total.

- **Two pools.** **Weekly** — 16 single-game challenges, of which a **calendar-deterministic 5-wide slice** is active each ISO week (so every player worldwide sees the same 5 on the same days); a fresh week resets weekly-unlock progress. **Lifetime** — 19 achievements that accumulate across all games (total deliveries, kills by type, best wave, play time, etc.). Each definition is `{ id, pool, name, desc, cur(s,l), text?, goal | tiers }`; `cur()` reads the per-game stats bag (`game.stats`) and the persisted lifetime counters (`Achievements.lifetime`), plus `game.wave` for the "reach wave N" ones.
- **Single-goal vs tiered (v3.0 P7/P8).** A **single-goal** entry (all 16 weekly + the 5 non-tiered lifetime) carries a `goal` and is **unlocked when `cur >= goal`** — one unlock, one toast. A **tiered** lifetime entry (14 of the 19) carries an ascending `tiers:[bronze,silver,gold,titanium,platinum,diamond]` ladder instead of a `goal`; `evaluate()` unlocks **every newly-crossed tier in one pass** (a big jump can pop several toasts at once, ascending, each naming its tier — "Recycling Magnate — Gold"), and the module persists the **highest tier index reached per id** in `lifetimeTiers{id:idx}`. `TIER_NAMES` + a warm-metal→white `TIER_COLOR[]` (kept in the vector-glow palette so a badge never reads as a hazard tell — FLAG A-1-a) live with `COLOR`.
- **Two lifetime counter kinds — SUM vs MAX (FLAG A-1-b).** Most lifetime counters are **sums** (running totals, `l.X++` at the event). Four tiered ones tier on a **personal-best MAX** instead — updated with `Math.max(...)` at the event, never `+=`: **Master of the Field** (`l.maxWave`), **No Powerups Needed** (`l.maxWaveNoPowerup` — best wave reached while `game.stats.powerupsPicked === 0`, updated at each wave-clear and **frozen** for the rest of that game the moment any powerup is picked), and the two P8 personal-bests **Salvage King** (`l.bestDeliveredGame`, most canisters delivered in one game) + **Field Sweeper** (`l.bestDebrisGame`, most Debris destroyed in one game — both `Math.max`'d at their event site as the per-game count climbs). The tier machinery is identical, only the counter's update rule differs.
- **Weekly rotation (deterministic, no randomness).** At launch the real UTC date yields an ISO week + week-year via `isoWeekYear()` (Monday-based, Thursday-of-week rule); the pool index is **`(isoYear × 52 + isoWeek) % 16`** and the active set is the 5 consecutive pool entries from that index, **wrapping** around the 16. Every player in the same calendar week gets the same 5.
- **The 16 weekly:** Scrap Runner (deliver 20 canisters/game), Heavy Hauler (a full 12-chain in one dock visit), Glass Cannon (reach wave 5 with no Health powerup), Satellite Buster (15 Debris/game), Hunter's Bane (wipe a whole 13-member Hunter line from one spawn), Pacifist Tow (5 deliveries in a row without firing), Close Shave (survive a Hunter hit under 10 HP), Shield Surfer (10 shield deflects/game), Small Ball (10 small saucers/game), No Scratches (clear wave 3 damage-free), Combo Collector (delivery combo 8+ in one visit), Speed Recycler (first delivery within 60 s), Powered Up (use all four drop powerups/game), Diamond Cutter (3 large Hunter cores/game), Waste Not (**v3.2 P3, repurposed in place** — finish a game with **no Hunters born from neglected scrap**, i.e. `s.hunterCoalesced === 0`; the id + pool slot are kept so the weekly rotation math is unchanged. Its old meaning, "zero canisters expired," auto-unlocked every game once decay was removed, so it was repurposed to a live challenge with the same "don't let your mess pile up" spirit — §2.5.1), and **Flawless Run (v3.0 P8** — clear **wave 8 or later** damage-free; a deliberately harder, late-wave counterpart to No Scratches, FLAG A-6-b, keyed off `s.flawlessLateWave` set in the wave-clear block).

### 2.18 High Scores (v3.6 P6)

A local top-10 table, its own `HighScores` module (defined right after `Achievements`) — deliberately built so a **future remote leaderboard can adopt it with no data migration**.

- **The wire shape.** Each record is `{ v, id, initials, score, wave, delivered, ts, build }`: `v` (1) versions the record independently of the store's own `v`; `id` (`crypto.randomUUID?.()`, falling back to a random hex string) is a **stable client-generated dedupe key** — a server can later accept an upload of the whole local table without double-counting; `ts`/`build` are provenance. A `playerId`/`name` field is purely **additive** whenever login lands — nothing in the existing shape has to change. The store itself is `{ v: 1, entries: [...] }`, sorted score-desc, sliced to `SCORES_MAX` (10).
- **Qualification.** `HighScores.qualifies(score)` — `false` for `score <= 0`; otherwise `true` if the table has fewer than `SCORES_MAX` entries, or `score` **beats** (strictly, not ties) the current 10th place.
- **Entry, at the `dying → gameover` seam (§2.7/§2.9).** A qualifying run arms `game.entry = { initials:[0,0,0], idx:0 }`; the gameover screen renders three big letter slots instead of the usual table/prompt. Up/down cycles a slot through `SCORES_CHARSET` (`A–Z`, `0–9`, space); left/right moves the cursor; confirm calls `commitEntry()`, which builds the record from `game.entry.initials` + `game.score`/`game.wave`/`game.stats.delivered`, hands it to `HighScores.add()` (insert, re-sort, cap, persist), remembers the new record's `id` in `game.lastScoreId` (the table-highlight key), and clears `game.entry`. Both input paths — keydown's normal branch and the gamepad's already-edge-detected `menuNavEdges()` — route through the same `entryInput(action)` dispatcher; a keyboard `e.repeat` guard (same idiom as the v3.5 P1 menu-nav fix) and the gamepad's pre-existing edge detection both stop a held key/stick from spinning the letter. **Deliberately not a menu screen** — `game.paused` stays false throughout, so the field keeps rendering behind it like the rest of the (unpaused) gameover screen.
- **The load-bearing guard.** The pre-existing keydown site that starts a new game on confirm from `gameover` gained `&& !game.entry`; entry-handling also **returns early**, before that check is reached, so a confirm that just committed the entry can't fall through and restart the run in the same keypress (§2.9).
- **Rendering.** `drawScoreTable(cx, topY, highlightId, scale = 1)` is the one renderer shared by both places a table appears: the gameover screen (`game.lastScoreId` highlighted in `COLOR.ach`) and a read-only **"High Scores"** row, opening a `"highscores"` screen (`menuHighScores`/`drawHighScores`) that reuses the same table with no highlight. A missing row (an empty or partial table) just draws a dim rank number — crash-free at any fill level. **Reachability (CS010 P4, §8b, FORK-4): the row moved off the system-menu root (`MENU_ROOT_SYS`) onto `MENU_OPTIONS`** — `Options > High Scores` — so it's now reachable from **both** the system menu (title/gameover, via Options) **and** the pause menu mid-game, which the old root-only placement never allowed; `menuHighScores` backs to `"options"` via `MENU_OPTIONS.indexOf("High Scores")`, and with one entry point there's no `scoreReturn` tracker (unlike the two-entry-point `achReturn` pattern). The trade-off: High Scores is now two keypresses from the title (O → Options → High Scores) instead of one.
- **Font size (CS010 P5, §8a).** `scale` multiplies every font size, the row pitch/header offset, AND the column x-offsets together (bigger glyphs need wider columns). The two callers deliberately diverge: **`drawHighScores()` passes `HS_TABLE_SCALE` (1.8)** — this screen has no competing content below the table, so the panel grows to fit (`menuPanel(1000, 560, "HIGH SCORES")`, table at `y + 120`, footer moved to `y + 497`, all hand-computed for 10 rows at the 1.8 pitch, not derived at runtime). **The gameover caller passes no 4th arg (stays at scale 1, unchanged)** — its layout is already tight against "PRESS ENTER TO PLAY AGAIN"/the O-hint below it (§8a's own math: 180% there would push row 10 to ≈ 747, off the bottom of a 720-tall viewport, through both prompts), and the fresh entry is already highlighted gold, so it doesn't need the bigger type to read. **Don't unify the two calls** — re-flowing the gameover screen to also afford 180% is a separate, un-scoped job (see STATUS.md playtest ask).
- **Persistence (`afd_scores_v1`).** A **third**, independent `localStorage` key, reusing the same guarded `storageOK()` try/catch idiom as `afd_settings_v1`/`afd_achievements_v2` — and touching **neither** of those two frozen keys. `HighScores.load()` tolerates a missing or corrupt payload (falls back to whatever was already in memory, never throws); `save()` swallows quota/privacy-mode/disabled-storage failures the same way.

> New constants: `SCORES_MAX` (10) and `SCORES_CHARSET` (`"A–Z0–9 "`) sit with the other v3.6 P6 additions near `DEATH_FLASH_ALPHA`; `GAME_VERSION` (`Major.Minor.Patch.Changeset`, "1.0.0.11" as of CS011 — CS010 P0 §7 — stamped into each record's `build` field) sits alongside them. The `HighScores` module + `entryInput`/`commitEntry` sit right after `Achievements`, before `powerTypesUsed`. `drawScoreTable`/`drawEntrySlots`/`drawHighScores` are near `drawAchRow`/`drawToasts`. `game.entry`/`game.lastScoreId` sit with `deathT`/`shake` on the game object, both reset in `startGame`.

### 3.1 Collision conventions
- All collisions are **circle vs circle** using wrap-aware `dist2` against squared radii. Polygon shapes are cosmetic.
- Pass order in `update()`: garbage pickup (during garbage update) → player bullets → targets; hostile bullets → ship → **chain nodes**; hazard bodies → ship (using a spread copy `[...arrays]` so `split()` pushes during iteration are safe); **hazard bodies → chain nodes** (spread copy, labeled `chainScan` loop, first hit wins per frame); saucer bullets → debris satellites.
- Entities are killed by setting `dead = true`; arrays are filtered once at the end of the frame. **Follow this pattern** — it avoids iterator invalidation bugs. (Exception: chain nodes are removed via `breakChain`/`chain.pop()`, never a dead flag — see 3.4.)
- Ship collision radius switches to `SHIELD_RADIUS` (26) whenever the shield is up. The shield does **not** extend to chain nodes (deliberate — see 2.10.1).
- Chain nodes use an effective radius of 7 vs hazard bodies and 9 vs hostile bullets.
- **Unshielded contact/bullet hits call `damageShip(amount, hazardX, hazardY)`** (v1.3) instead of the old instant-kill `killShip()`: it subtracts source-specific HP, applies knockback + a 1.0 s hit-stun, and only routes to `killShip()` (game over) at 0 HP. Shielded contact deflects/blocks instead (no damage). Because `damageShip` self-guards on shield/hit-stun/dead, the hazard-vs-ship loops no longer `break` after the first hit — a shielded ship still deflects every overlapping hazard, and an unshielded one absorbs exactly one hit per stun window (§2.12).

### 3.2 Rendering conventions
- `drawPoly(points, x, y, angle, color, closed)` + `glowStroke(color, width, blur)` produce the vector look via `shadowBlur`. New entities should define local-space point arrays and reuse these — don't invent per-entity draw pipelines. Canisters (free and chained) share one renderer, `drawCanister()`.
- `drawLink()` skips any link whose wrap-aware length exceeds 120 px — this prevents chain segments from streaking across the screen at the wrap seam. Keep this guard if touching chain rendering.
- `shadowBlur` is the main perf cost. If adding many simultaneous entities (>~80), consider a global glow toggle. The salvage feature adds up to ~12 chain nodes + ~15 free canisters at once; watch frame rate in garbage-heavy late waves.
- **The HUD is a `glowStroke` client like everything else (CS009 P0–P6).** The old hull/shield/cargo/powerup `fillRect`/`strokeRect` bars are **gone** — every HUD ring/arc is drawn via `drawRingArc()`, which routes through `glowStroke()` like any world entity. The surviving HUD fills are `drawText` (which is `fillText`) and — as of CS010 P3 — the **low-health corner glow** (four radial-gradient corner fills), kept as a deliberate, named exception (see below).
- **No fills, except bullets/particles/HUD text — plus two deliberate exceptions.** *(CS012 P2 dropped the SCOOP pip row exception — the pips are gone, replaced by `drawRingSegments()`'s stroke-only wedges, which add no fill. The count previously held at three across CS010: P1 retired the Scoop capture box, P3 added the corner glow.)* **(1, v3.6 P5)** The **death-spectacle white flash** (§2.7/§2.9) is a full-viewport `ctx.fillRect` at decaying alpha, drawn in screen space after the world on the first frames of the `dying` state — a transient screen-wash, not an entity fill. It is a **fill by design, not by convenience** (FORK-3, Paul signed off): a peripheral-vision alarm is low-frequency, large-area and **edgeless** — that is exactly what lets it register out of the corner of the eye without competing for focus. A `glowStroke` corner arc would be the opposite (a line: high-frequency, small-area) and would read as a *third* red HUD element next to the two red rings — the visual noise §2.12 exists to avoid — besides being more expensive (`glowStroke` is `shadowBlur`, the renderer's hot spot; four blurred arcs would burn it in the busiest frames). - **New sound:** add an `AudioSys` method; one-shots follow `fire()`, loops follow `thrust()`.
- **New cargo-adjacent mechanic:** work through the existing chain functions — score through `addScore`, break through `breakChain`, convert nodes with `Garbage.fromNode`.

### 3.4 Chain physics contract (v1.1) — read before touching `updateChain`

The tow chain is the only physics system in the game that is *not* the standard entity pattern. Its rules:

- **Node shape:** plain objects `{ x, y, px, py, spin, spinRate, mass }` in the `game.chain` array, ordered head (nearest ship) → tail. `px/py` are the previous-frame verlet positions; implied velocity is `(x−px, y−py)`. **`mass`** (v1.4/F5) is copied from the picked-up `Garbage` (default 1.0) and drives the handling penalties — **any code that creates a chain node must set `mass`**, or `chainMass()` will `NaN`. There is no `dead` flag and no class — nodes are created inline at pickup and removed only by `breakChain(i)`, `scatterChain()`, or `chain.pop()` at the dock.
- **Integration order per frame** (inside `updateChain`): (1) verlet integrate all nodes with damping `0.18^dt`; (2) measure the **pre-constraint** stretch of the first link and apply the momentum tug to the ship; (3) run `CHAIN_ITER` (4, was 3) relaxation passes of position-based distance constraints (link 0: node takes 100% of the correction, the anchor is kinematic; links 1+: 50/50 split between neighbor nodes); (4) `wrapNode` every node.
- **Why tug-before-constraints:** the pre-constraint stretch is the honest measure of how much the cargo resisted the ship this frame. Applying a spring force *after* constraints (when stretch ≈ 0) does nothing; applying stiff springs *instead of* constraints oscillates. Don't reorder these steps.
- **Wrap handling:** `wrapNode` shifts `px/py` by the same amount as `x/y` so implied velocity survives the teleport — this is why the chain doesn't explode at screen edges. All link deltas (constraints *and* rendering) go through `shortDelta`. Any new code measuring between two nodes must do the same.
- **Constraints are equality (rod-like), not rope-like** (they also correct compression). This is what produces the serpentine follow instead of nodes bunching on the ship when it decelerates. If you change to stretch-only constraints, expect the snake feel to disappear.
- **Cargo penalties live in `Ship.update`**, not in the chain code: `thrustMul` and `maxSp` are computed from **`chainMass()`** (Σ node.mass, v1.4/F5), and **turn rate now flows through `shipTurnRate()`** (CS010 §10b/§3) — `SHIP_TURN × settings.shipTurnScale / (1 + chainMass() × CARGO_TURN)`, the single source both `Ship.update` turn sites read, so they can't diverge. The momentum-tug `massFactor` in `updateChain` uses the same `chainMass()` sum. Before v1.4 the thrust/speed penalties used `game.chain.length` (a count); for all-mass-1.0 cargo the sum equals the count, so v1.3 balance is preserved. `CARGO_TURN` ships at **0** (the cargo turn penalty is dormant — the proposed "inertia" fix, Paul's one-number call to enable; at 0, `shipTurnRate()` is exactly `SHIP_TURN × scale`, mass-independent). `settings.shipTurnScale` is a **player** rotation-speed preference (0.5–1.5, persisted in `afd_settings_v1`), not a balance value, and is deliberately **not** reset by Controls' "Return to Defaults" (FLAG-10a). If you add further cargo effects (e.g. shield drain), put them in `Ship.update` alongside and scale off `chainMass()` too.
- **Stability envelope:** re-validated for v3.4 P1 at dt ≤ 0.05 (the main-loop clamp), **`CARGO_CAP_MAX` (**24**, up from 20) nodes, `CHAIN_ITER` (4) constraint iterations** (unchanged — re-validated, not assumed). Headless stress (`scratchpad/test-p6.js` (E)): a **24-node** chain under hard thrust-flips across a wrap, 900 frames, shows no NaN, no velocity explosion, and worst-case link stretch ≈**4.11 px** on a 20 px link — under the ~5 px budget, so `CHAIN_ITER` did **not** need to bump to 5 (measured directly: at 4 iters 4.11 px, at 5 iters 3.67 px — both comfortably stable, 4 was kept). This is barely above the prior 20-node measurement (~4 px), consistent with B-8's iteration count already carrying headroom past 20 nodes. The worst link is link 0 (anchor→head, dominated by ship speed), not the tail, so corrections propagate the full length — the chain doesn't sag. If you lengthen the chain or the link length further, revalidate the same way (see 5.4 rule 7) before shipping; the `drawLink` 120 px wrap-skip still reads correctly at 24 nodes. **CS010 P2 re-ran this (FLAG-3a) after retuning the mass coefficients** (`scratchpad/test-cs010-p2.js` (D)): at the documented 1/60 methodology the worst-case stretch is **still 4.11 px** — byte-identical to the pre-CS010 build (delta 0.000), because the tug's 1.4 cap is unchanged and the kinematic stress isolates the constraint solver; under a realistic *faithful* stress (real `Ship.update`, tug feeds back) at the true 0.05 clamp the retune is if anything marginally *stabilizing* (worktree ≤ pre-CS010 at every mass — 8.17 vs 8.33 px at mass 24, 11.61 vs 12.61 px at the light-and-fast mass 6 — a stronger tug is a stronger restoring force). **Record-correction:** the 4.11 px figure is measured at dt = 1/60, not the 0.05 clamp; the same deliberately-unphysical over-stress velocity (v=420 ⇒ 21 px/frame) at 0.05 gives ~14.8 px on *both* builds — a big-timestep artifact, still no NaN/blowup — so "dt ≤ 0.05" above bounds the sim's validity, not the number's measurement dt. ## 6. Bug Watch List (v1.1)

Areas most likely to hide issues, worth checking during playtests:

- **Shield-deflection edge cases** — deflecting a rock into a wall of other rocks; deflecting at exactly the moment energy hits zero. New in v1.1: deflecting a rock directly into your own tow chain (working as designed, but verify it feels fair).
- ~~**Respawn deadlock**~~ — *resolved by F2 (v1.3): there is no respawn anymore, so this can't occur.*
- **Knockback vs. tow chain (v1.3)** — knockback shoves the ship away from a hazard, which can drag a long chain tail through *other* hazards or fling it across the wrap seam. The chain is wrap-aware and severs per its existing rules, but verify a hit never yanks the ship somewhere that instantly severs the whole haul unfairly.
- **Knockback into a second hazard (v1.3)** — the 1.0 s hit-stun means being knocked straight into another rock is safe *during* the window; verify it doesn't feel like the ship pinballs helplessly once the window expires in a dense field.
- **Saucer sound leak** — the warble is stopped on saucer death/exit and on game over; verify it can't persist if a saucer dies the same frame the ship does.
- **Performance (raised in v1.4, again in v1.6; revisited v3.3 P4, v3.4 P1)** — heavy `shadowBlur`; the salvage feature already raised the entity ceiling, and F3's guaranteed per-tier drops raise the *free-canister* count far higher (a fully-cleared Debris large lineage emits **52** as of v3.4 P1's `DEBRIS_GARBAGE` 3→4 bump; a fully-cleared Hunter lineage — of **any** origin as of v3.3 P4 9b — now emits **18**, down from a flat-rate 66 now that v3.4 P1 made `HUNTER_GARBAGE` a per-tier table, so Hunter lineages are no longer the larger of the two sources). **v3.3 P4 reintroduced single-garbage decay** (`GARBAGE_DECAY` ≈ 22 s, singles only), which caps how much standing salvage accumulates even with every Hunter emitting again; clumps persist. Render is viewport-culled (`onScreen` in `drawEntity`), so free-canister count doesn't drive *draw* cost; the O(n²) `coalesceGarbage` pass is the watch item (FLAG D-2 — cheap pair tests, survivable, and coalescence compacts `n` downward; v3.4 P1's net drop-volume change is a modest reduction, not an increase, so this doesn't get worse). Watch frame time at wave 10+; if it dips, a global glow toggle (§3.2) is the lever, then the drop-volume constants (`DEBRIS_GARBAGE` / `HUNTER_GARBAGE`), then (only if a stress test shows it) a spatial hash for the coalescence pass — explicitly out of scope until then.
- **Hunter shield-kill at point-blank (v1.6)** — destroying a medium Hunter on the shield spawns 3 smalls right at the shield edge; they scatter outward for 0.6 s before homing, so it *shouldn't* be an instant point-blank swarm, but verify it feels fair (a fair "spend energy, but now deal with the smalls" trade, not a death trap) — this is the old wedge tension carried forward.
- **Hunter early-game feel vs. escalation (v1.6)** — the whole family moves at `HUNTER_FLOOR_FRAC` (0.58) of full speed at wave 1. Verify the large core's drift and any split children genuinely feel *gentle* in the opening waves (not still-too-fast) and that both tier-climb and wave-climb read as a smooth escalation, not a cliff. `HUNTER_FLOOR_FRAC` (family-wide early calm) and `RAMP_WAVES` (whole-game pace) are the knobs.
- **Garbage density & the re-tuned coalescence economy (v3.3 P4, SUPERSEDES the permanent-salvage ask)** — single decay is back and the whole coalescence economy was re-tuned around it. The open questions: with `GARBAGE_DECAY` (≈ 22 s), `GARBAGE_COALESCE_DELAY` (≈ 3.0 s), and `GARBAGE_MAGNET_RANGE` (≈ 180) all moved together, do clumps still form *at all* before singles age out — or does decay now starve coalescence (if so, cut the delay or lengthen decay)? Post-9b (every Hunter drops again), does the field **silt up** or does decay hold it through late waves? Does the reddening clump hull read as "shoot me" **or** now "scoop me" (both are correct)? And is the **scoop vs. shatter** choice live — do you sometimes shatter for a lossless comeback rather than greedily scoop and waste the overflow? These are the top v3.3 P4 retune asks; the `nextWave` `min(3+wave,9)` count / `SPAWN_MAX_DIST` are the direct levers if the late-wave lull persists (§9 preamble).
- **Powerup drop rate & Magnet feel (v1.7/F6)** — `POWERUP_DROP_CHANCE` (10%) and the ambient `POWERUP_HEALTH_GAP` (18–26 s) are first-pass, no playtest data. Verify drops feel *earned but not stingy* (not so rare they never matter, not so common they stop feeling special), and that the Magnet's pull reads as a satisfying sweep rather than jittery orbiting (levers: `MAGNET_PULL` up-front accel, `MAGNET_DAMP` in-range damping — lower `MAGNET_DAMP` toward 0 to make it settle harder, or drop `MAGNET_PULL` if garbage zips/overshoots). Also confirm the five icons read distinctly at a glance and the HUD duration row is legible.
- **Controller feel & audio unlock (v1.8/F7)** — the binding *logic* is headless-verified, but deadzone comfort, whether right-trigger *or* right-bumper is the right shield button (a gap-fill, not spec'd), and general button-mapping feel can only be judged with a real controller in a browser — **the top v1.8 playtest ask.** `GP_DEADZONE` (0.25) is the one feel knob. Also verify: does the D-Pad-vs-stick split feel natural; does Start pausing/unpausing read cleanly; and — since browsers don't reliably treat gamepad input as an audio-autoplay gesture — does a controller-only start produce sound, or is one keyboard press still needed to unlock audio? (If the latter, that's a known browser limitation, not a bug.)
- **Menu feel & first-time persistence (v1.9/F8)** — the menu *logic* is headless-verified (40 assertions), but three things need a real browser: (1) does menu navigation feel natural on **keyboard and gamepad** (slider granularity at `VOL_STEP` 0.1; edge-detected one-press-one-move gamepad nav vs. wanting held auto-repeat); (2) does a rebound control take effect **immediately** in live play, and are the two flagged quirks acceptable — a gamepad rebind **dropping that action's stick axis**, and **no conflict detection** when rebinding to an already-used key; (3) **do settings actually persist across a real reload** — this is the first feature to write `localStorage`, which is absent in the artifact sandbox, so real persistence is unverified (worst case in the sandbox is "no persistence," never a crash — access is guarded). See §2.16.
- **Chain tug oscillation** — `CHAIN_TUG 26` is stable in headless tests but hasn't been feel-tested against every input pattern. Rapid thrust-flip-thrust with a full chain is the stress case; if the ship judders, lower `CHAIN_TUG` before touching the constraint code.
- ~~**Dock/respawn interaction**~~ — *moot post-F2 (v1.3): no respawn. `killShip()` still calls `scatterChain()`, so a dead ship never keeps a chain.*
- **Wave-transition cargo** — the chain persists across waves (deliberate) but new large debris satellites spawn ≥220 px from the *ship*; a long chain tail can extend beyond that. Verify a wave spawn can't instantly sever a fresh chain unfairly often.
- **Delivery combo reset radius** — combo resets at dock radius+40. Verify a player orbiting the dock edge can't accidentally keep/lose the combo in a confusing way.
